## 一、一句话系统架构

```
信息源（博主/新闻/RSS）
     │
     ├──────────────────┐
     ▼                  ▼
events/{日期}.md   00-宏观传导框架.md
（单次写入）       （宏观框架更新）
     │                  │
     └──────┬───────────┘
            ▼
      positions/{资产}.md
      （贝叶斯更新 + 事件链）
            │
            ▼
      23-贝叶斯更新日志.md
      （聚合索引）

━━━ 自动化监控（独立于手动流程）━━━
  cron 每日10:00
  每日风险信号汇总
  → 读全部持仓+事件+宏观
  → 发现多源聚合信号就预警
  → 完整报告存 events/*-risk-summary.md
  → ⛔ 不写 positions/，不改 P 值
  → Hermes 看完报告后再手动判断是否补事件链

━━━ 定期验证（耦合进决策回看·每两周）━━━
  每14天 14:00，自动运行 p_price_verify.py
  → 读每个持仓的P值 + 实时价格
  → 比对方向是否一致
  → 背离则标记 ⚠️
  → 只出报告不改数据
```

### ⚠️ 路径找不到时的操作协议

**当任何技能/脚本/文件路径找不到时，不准猜测/不准试错，按此顺序查：**

```
技能找不到（skill_view报错）：
  ① skills_list → 看技能名是否正确拼写
  ② search_files(target="files", pattern="*关键词*", path="~/.hermes/skills/") → 全目录搜
  ③ 看桌面路径快照: cat ~/桌面/系统路径快照.txt → 找原始路径
  ④ L3外部记忆: cat ~/.hermes/external_memory/16-B站视频处理管线.md → 有流程备份
  ⑤ 还找不到 → 直接告诉用户「我找不到XX文件了，可能路径断了，桌面路径快照.txt里有原始路径」

脚本找不到（执行报错）：
  ① ls ~/.hermes/scripts/ → 看目录下有什么
  ② search_files(target="files", pattern="*关键词*", path="~") → 全盘搜
  ③ 桌面路径快照.txt → 有完整路径备份

核心：搜，不猜。猜新路径=大概率断引用=又要花半小时debug。
```

**路径快照文件：** `~/桌面/系统路径快照.txt`（2026-05-24固化）— 所有关键路径的一次性完整记录。

## §0 最高铁律：遇事先读文档，不闷头试错

**任何时候遇到不确定的流程，优先级：**

```
① skills/（技能目录）→ load技能看步骤 → 技能文件是唯一权威流程
② INDEX.md（本文档）→ 导航到正确文件
③ hindsight_recall（L2语义搜索）
④ 问用户
```

**不要凭记忆试错。** 本系统经验证明：凭记忆做 → 大概率做错 → 用户纠正 → 耗时翻倍。

### ⚠️ 路径铁律：不动路径是系统稳定性的基石

**绝对禁止：**
- ❌ 重新分类/移动桌面文件（`~/桌面/`下的任何文件）
- ❌ 重命名凭证文件（`~/桌面/凭证/`）
- ❌ 移动脚本文件（`~/桌面/脚本/` 或 `~/.hermes/scripts/`）
- ❌ 改变skills目录结构（`~/.hermes/skills/`）
- ❌ 把文件从桌面搬到.hermes，或从.hermes搬到桌面

**为什么：** cron、桌面快捷方式、其他脚本都硬编码路径。动一个路径 = 断一串引用。之前多次因为归档分类搞坏路径被骂。

**所有文件维持现状。** 如果要新增文件，放 `events/` 或 `positions/` 目录下。

## §1 三平台监控系统

> **详细实操手册 → `00-网站内容抓取总纲.md`**（精确到脚本路径/curl命令/cookie流程/铁律/崩溃恢复）
> 以下为本节概览，实操细节以总纲为准。

### ⚠️ 通用前提条件

**CDP Chrome必须已运行**（端口9222）：
- Cron每天23:03自动启动：`google-chrome --remote-debugging-port=9222 --user-data-dir=/tmp/chrome-debug`
- 如果挂了 → 手动启动：`bash ~/桌面/一键监控.sh`（该脚本自动启动CDP Chrome）
- 如果CDP Chrome被OOM杀了 → 先跑`内存急救.sh`释放内存

**每个平台都需要用户登录一次**（登录态cookie存桌面）：
- 凭证路径：`~/桌面/凭证/{平台}_cookie_最新.txt`
- cookie有效期：抖音/知乎~30天，B站7-30天
- 过期时API返回限流/未登录 → 需要用户在CDP Chrome上重新登录

### 平台1：抖音（最稳定，直接跑脚本）

**方法：直接跑监控脚本**
```bash
# 横杠版（唯一有效版本）：
python3 ~/.hermes/scripts/douyin-monitor.py --headless
# 或双击：~/桌面/一键监控.sh（三平台全部跑）
```

**脚本说明：**
- 脚本：`~/.hermes/scripts/douyin-monitor.py`（横杠版，硬编码14个sec_uid）
- cron运行：`~/.hermes/scripts/douyin_monitor.py`（下划线版，读INDEX.md）
- ⚠️ 注意有二个版本：横杠版(有效) vs 下划线版(可能旧的sec_uid)
- 输出：`~/桌面/每日视频总结/{日期}_监控报告.md`

**已知问题：** 如果API返回空 → cookie过期 → 需要在CDP Chrome重新登录抖音

  浏览器备用方案（API限流时）：

模块C→E→F→G：转录分析（按bilibili-video-pipeline技能操作）

**完整操作流程（如技能文件`bilibili-video-pipeline`所述）：**

## §2 核心工作流程

### 流程A：处理博主新信息（日常最多的工作）

```
收到博主新视频/文章
     │
Step 1 信息分层
     ├── L1 事实/数据（具体数字、公告、可验证事件）
     │   → 写进 events/{日期}.md + thinkers/{博主}.md
     ├── L2 洞察/逻辑（推理链、因果关系、时间窗口）
     │   → 写进 thinkers/{博主}.md + 作为贝叶斯推理依据
     └── L3 框架/原则（可复用投资原则，如"越靠近GPU越值钱"）
         → 还要嵌入 00-宏观传导框架.md
     │
Step 2 影响哪些持仓？
     ├── 在 events/{日期}.md 里列全（持仓 | 方向 | 逻辑）
     └── 如果来自博主 → 在 thinkers/{博主}.md 也加一列
     │
Step 3 贝叶斯更新
     ├── 打开 positions/{资产}.md
     ├── 写：以前P(上行)=X% 因为 A 理由
     ├── 写：新信息改变了什么 → 先验变化描述
     ├── 写：贝叶斯更新（因子 | 调整 | 依据）
     └── 写：合计 ±Xpp | P(上行) X%→Y%
     │
Step 4 更新聚合索引
     └── 23-贝叶斯更新日志.md 追加一行 + 更新当前概率总览
     │
Step 5 验证互锁（见第四节）
     └── 检查所有受影响文件是否已更新
     │
Step 6 决策记录（用户执行了买卖时）
     ├── 打开 决策日志/{日期}.md
     ├── 写：类型/资产/金额/逻辑/P值当时/信息源触发
     └── 关联：在对应 positions/{资产}.md 加备注

Step 7 推给用户
```
### 流程F：问资产审核

**每次问任何资产，必须完整走这套：**

```
⚡ 第零步：定性分类（前置判断）
    ├── 🥇 主力方向：研究透了/看得懂 → 敢上量，三关从严
    └── 🥈 兜底方向：研究不透/看不懂 → 买便宜货，三关可松
    ├── 决定了后面三关的松紧度和估值方法
    └── ⚠️ 不确定用什么估值方法 → 先问用户

Step 1: 🌍 宏观层 → 读00-宏观传导框架.md
Step 2: 🏭 中观层 → 行业/板块/资金流向
Step 3: 🔬 微观层 → P值+仓位+三关结果

   🎯 估值方法：不同行业用不同方法
       ├── 🥇 主力方向/研究透了 → 用PEG/合理估值
       ├── 🥈 兜底方向/研究不透 → 用PE/估值回归
       └── ⚠️ 不确定用哪个 → 问用户"这个该用什么估值？"
       └── 当前已知：澜起科技=Fabless芯片设计→PEG（用户纠正过）

Step 4: 🎯 3选1动作 → 不动/加仓/减仓

⚠️ 结论必须落脚到动作，不准出现"等等看""待跟进"
```

**三关审核完整流程 → 详见 `asset-quick-lookup` skill → `references/valuation-methodology.md`**
（含：三关逐项过法、不同类型资产跳过规则、仓位权重公式、资金分配优先级）

## §3 已知故障模式与解决方案

| 问题 | 原因 | 解决 |
|:----|:-----|:-----|
| B站API返回-799 | IP限流 | 等10-30秒重试，或用浏览器看空间 |
| B站API返回isLogin:false | cookie过期 | 重新登录B站+提cookie |
| 抖音API返回空 | cookie过期 | 重新登录抖音+提cookie |
| CDP Chrome连不上9222 | Chrome被OOM杀了 | 重新启动：`bash ~/桌面/一键监控.sh` 或cron启动 |
| 知乎脚本返回0新但有最新时间戳 | 状态文件已记录 | 正常，不是脚本坏了 |
| 知乎脚本全部返回401 | cookie没传进fetch | 检查cookie参数是否传入fetch函数 |
| 向量库挂了（hindsight） | OOM或内存不足 | 先跑`内存急救.sh`，不影响skills和L3文件 |
| 转录失败（DNS error） | DNS临时故障 | 重试一次即可 |
| 转录失败（容器不存在） | VideoCaptioner没启动 | `docker start videocaptioner` |

### ⚠️ 向量库（hindsight/L2）挂了时的工作铁律

**向量库仅存会话上下文索引，不存技能和L3文件内容。挂了不影响：**
- ✅ `~/.hermes/skills/` → 技能文件（总是可读）
- ✅ `~/.hermes/external_memory/` → L3文件（总是可读）
- ✅ 桌面文件（`~/桌面/`）→ 脚本+凭证（总是可读）
- ❌ 会丢失：之前会话的进度、中间分析结果、任务状态

**向量库挂了后正确做法：**
1. 先读skills/ → 找到对应流程技能
2. 再读L3文件 → 找到当前状态
3. 直接从当前步骤继续，不回溯之前会话

## §7 铁律（不可违反）

1. **没看过/没转录的内容不准概括。** 宁可写「没看过」也不能编。
2. **贝叶斯更新必须有"以前怎么想→现在怎么想"的信念变化说明。** 禁止只写"利好+3pp"。
3. **修正不覆盖，追加修正。** 判断错了→追加一条修正记录，保留原始记录。
4. **每项资产一定要关联到宏观层。** 不能说"涨了"就结束，要说清楚为什么。
5. **不自作主张改配置。** 改配置前备份+通知小龙虾，改完10分钟互相确认。
6. **系统卡死第一反应：** `bash ~/桌面/脚本/内存急救.sh`
7. **搞不定的事不硬磕。** 3次工具调用没结果→问用户或找更强模型。
8. **审核资产必用估值方法论。** 用户说"看看/审核XX"时，必须走流程F：三层+三关+3选1。
9. **图片不能看就用OCR。** 用`image-ocr`技能提取文字。
10. **绝对不动路径。** 桌面/脚本/凭证/skills的路径全部固化，不能移动/重命名/重新分类。
11. **向量库挂了不影响技能使用。** skills/和L3文件在磁盘上，随时可读。
