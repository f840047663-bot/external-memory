# 备份方案总览与恢复手册

## 一、备份体系架构

```
┌──────────────────────────────────────────────────────────┐
│                     Hermes 备份体系                        │
├──────────────┬──────────────────┬────────────────────────┤
│     层       │    存储位置       │      同步方式           │
├──────────────┼──────────────────┼────────────────────────┤
│  L1 内部记忆   │   GitHub仓库     │  每30分钟自动同步       │
│  (MEMORY.md  │  external_memory/ │  (crontab触发,         │
│   +USER.md)  │  memories/       │  cp到repo→commit→push) │
├──────────────┼──────────────────┼────────────────────────┤
│  L1 + L2     │   百度网盘        │  每3天自动上传           │
│  (系统规则+   │  /Hermes备份/    │  (Hermes cronjob,       │
│   向量库)    │                  │  分卷20MB×N个上传)      │
├──────────────┼──────────────────┼────────────────────────┤
│  L3 外部知识库 │   GitHub仓库     │  每30分钟自动同步         │
│  (全部MD文件) │  external_memory/│  (已稳定运行多时)        │
└──────────────┴──────────────────┴────────────────────────┘
```

## 二、两个备份怎么用

### 场景A：只恢复系统规则（最快）→ 用GitHub

**适合：** 换新AI、Hermes崩了但技能和习惯还在、只需要规则不用向量库

```bash
# 克隆仓库（如果还没有）
git clone https://github.com/f840047663-bot/external-memory.git ~/桌面/Hermes外部记忆

# 拉最新
cd ~/桌面/Hermes外部记忆 && git pull

# 把L1放到Hermes
cp memories/MEMORY.md ~/.hermes/memories/MEMORY.md
cp memories/USER.md ~/.hermes/memories/USER.md

# 重启Hermes
hermes start
```

**能恢复的东西：**
- ✅ L1系统铁律（300+条规则、用户偏好、工作流）
- ✅ L3外部记忆（全部文档、持仓分析、博主档案）
- ✅ 62个技能备份
- ❌ L2向量历史（不在此处）

### 场景B：完整恢复（连历史记忆一起）→ 用百度网盘

**适合：** 系统全损、换电脑、需要连向量库一起恢复

```bash
# 1. 从百度网盘下载最新备份
# 先看看有哪些备份
bypy ls /Hermes备份/ -t 60

# 下载最新的目录
bypy downdir "/Hermes备份/20260522_XXXXXX" ~/hermes-restore -t 120

# 2. 合并分卷
cd ~/hermes-restore
bash 合并.sh   # 生成 backup.tar 并解压

# 3. 停服务
hermes stop
kill -9 $(pgrep -f hindsight) 2>/dev/null; sleep 2

# 4. 恢复 L1
cp -r memories/* ~/.hermes/memories/

# 5. 恢复 L2（向量库）
cp -r pg0/* ~/.pg0/

# 6. 重启
hermes start

# 7. 验证
diff ~/.hermes/memories/MEMORY.md ~/hermes-restore/memories/MEMORY.md && echo "L1 OK"
curl -s localhost:9177/v1/default/banks/hermes/stats | head -3 && echo "L2 OK"
```

### 场景C：新AI从头接手

**适合：** 换了全新的AI，什么都不知道

```bash
# 第1步：读GitHub仓库
git clone https://github.com/f840047663-bot/external-memory.git
cd external-memory

# 第2步：读 INDEX.md（总导航）
cat INDEX.md

# 第3步：读 L1（系统铁律和用户画像）
cat memories/MEMORY.md
cat memories/USER.md

# 第4步：读 skills/ 目录（全部技能备份）
ls skills/
# 每个技能都是一个独立流程，按需加载

# 第5步：如果需要历史向量记忆，从百度网盘恢复
```

## 三、GitHub上的文件结构

| 文件/目录 | 说明 |
|:---------|:-----|
| `INDEX.md` | **总导航** — 新AI/新人第1份必读文件 |
| `memories/MEMORY.md` | L1 — 系统铁律、工作流、所有规则 |
| `memories/USER.md` | L1 — 用户画像、偏好、习惯 |
| `备份存档/内部记忆/` | L1按时间戳的历史版本（含 latest 标记） |
| `skills/*.md` | 全部62个技能备份（Hermes skills的备份） |
| `备份与恢复手册.md` | 本文件 |
| `00-*.md` 至 `99-*.md` | L3外部知识库全部文档 |
| `config.yaml.backup` | Hermes配置文件备份 |
| `scripts/*.py` | 自定义脚本（监控/备份） |

## 四、百度网盘上的文件结构

| 文件 | 说明 |
|:----|:-----|
| `part_00` ~ `part_N` | 分卷包（每卷20MB），N个文件组成完整备份 |
| `合并.sh` | 合并脚本：`bash 合并.sh` 一步到位 |

分卷合并后得到 `backup.tar`，解压后包含：
```
memories/MEMORY.md       ← L1 系统规则
memories/USER.md         ← L1 用户画像
pg0/                     ← L2 向量数据库（约300MB）
```

## 五、快慢对比

| 恢复方式 | 速度 | 能恢复什么 | 需要什么 |
|:--------|:----|:----------|:--------|
| GitHub（L1+L3） | 秒级 | 规则+文档+技能 | git clone/pull |
| 百度网盘（L1+L2） | 分钟级 | 规则+向量历史+全部 | bypy授权+网盘空间 |

**推荐策略：** 先拿GitHub的L1+L3（秒级恢复核心功能），百度网盘的L2有需要再下。

## 六、定时任务一览

```bash
# ⏰ 每30分钟 — L1+L3同步到GitHub（system crontab）
*/30 * * * * cp ~/.hermes/memories/*.md ~/.hermes/external_memory/memories/ && cd ~/.hermes/external_memory && git add -A && git diff --cached --quiet || (git commit -m "auto" && git push)

# ⏰ 每3天 — L1+L2上传到百度网盘（Hermes cronjob: "L1L2百度网盘备份"）
# 行为：打包→分卷→逐个上传到 /Hermes备份/{日期}/ → 清临时文件
```

## 七、首次设置（已做完，仅记录）

```bash
# 1. 装 bypy
pipx install bypy

# 2. 授权（只需一次）
bypy quota   # 提示打开链接→授权→粘贴授权码

# 3. token 存 ~/.bypy/bypy.json（30天有效，自动refresh）

# 4. 测试
bypy quota -t 60   # 显示配额即OK
```

## 八、注意事项

- 百度网盘 bypy 国内网络较慢，命令加 `-t 60`（超时60秒）
- 大文件分卷上传（每卷20MB）
- GitHub版本始终有最新的L1+L3，百度网盘只有每3天的快照
- L2向量库约300MB，只有百度网盘上有
- 百度网盘总空间2TB，已用435GB，空间充裕
