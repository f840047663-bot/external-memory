## 本地模型 vs API 评估（2026-05-12）

大好人的评估结论：

| 项目 | 结论 |
|:----|:------|
| 当前方案 | DeepSeek V4快速版，月费¥50-60，够用 |
| 本地替代 | Qwen3-27B量化≈V4快速版，持平或略强 |
| 硬件成本 | P40 ¥2000-3000，但机房退役卡有翻车风险 |
| 电费 | 250W×24h = ¥108/月，一年≈¥1300 |
| 首年总成本 | 硬件¥3000+电费¥1300 = ¥4300 > API年¥720 |
| 量化损失 | 27B跑P40必须4-bit量化，体验打折扣 |
| 维护 | 驱动/CUDA/爆显存/模型下载，费时间 |
| API优势 | 白嫖模型升级，永远用最新版 |

**结论：** 当前API方案最优。以后如果：
- 硬件价格大幅跳水（比如¥500能买到同级别卡）
- 本地模型质变（比如开源模型能打平V4旗舰）
- 社区风向变了

## [L2] Hindsight 向量库 — 三层记忆的RAM层

> ⚠️ **重要：** 这不是替代L1的工具。L2是L1满了以后的溢出层，相当于CPU的扩展RAM。
> 详细架构说明见 skill:memory-management → 「3层记忆架构」。

### 一、这是什么？

Hindsight 是一个运行在本地的向量记忆服务。它用 PostgreSQL 存数据，用 AI 模型把文字转成向量（语义指纹），这样我可以用"意思"来找记忆，不需要精确关键词。

**L1/L2/L3 关系（谁依赖谁）：**

```
┌─────────────────────────────────────────┐
│  Hermes AI Agent（我）                   │
│                                          │
│  L1 = 内部记忆（memory tool, 2200字）    │ ← 我自带，不需要Hindsight
│      ↓ 满了就迁到这里                    │
│  L2 = Hindsight 向量库（端口9177）       │ ← 需要 daemon 运行
│      ↓ 完整备份在这里                    │
│  L3 = 外部MD文件夹（./external_memory/） │ ← 纯文件，永远可靠
└─────────────────────────────────────────┘
```

### 二、正常运行时的样子

1. **daemon 跑在后台**，监听 localhost:9177
2. **嵌入模型** all-MiniLM-L6-v2 已加载到内存（约1GB RAM）
3. **PostgreSQL** 内嵌自动启动，存向量和文本
4. **LLM** Qwen2.5-7B-Instruct（硅基流动免费API），用来从文字中提取事实
5. 平时没人在 L2 上读写时，它几乎不消耗 CPU

### 三、启动脚本（搞清楚这段）

文件路径：`~/桌面/restart_hindsight_l2.sh`

脚本做了这些事：
1. 杀旧进程（占 9177 端口的）
2. 激活 Hermes 的 Python 环境（`.venv`）
3. 设置一堆环境变量，然后启动 `hindsight-api` 后台运行

**关键环境变量（改配置时不要搞错）：**

| 变量 | 值 | 为什么 |
|:----|:---|:-------|
| `HF_ENDPOINT` | `https://hf-mirror.com` | HuggingFace 国内镜像 |
| `HF_HUB_OFFLINE` | `1` | **必须！** 离线使用缓存模型，否则启动会卡死 |
| `HINDSIGHT_API_EMBEDDINGS_LOCAL_MODEL` | `all-MiniLM-L6-v2` | 嵌入模型，已经在缓存里 |
| `HINDSIGHT_API_LLM_MODEL` | `Qwen/Qwen2.5-7B-Instruct` | 硅基流动免费模型，用来提取事实 |
| `HINDSIGHT_API_LLM_BASE_URL` | `https://api.siliconflow.cn/v1` | 硅基流动 API 地址 |
| `HINDSIGHT_API_RETAIN_MAX_COMPLETION_TOKENS` | `8000` | 默认64000超了Qwen的32768限制 |
| `HINDSIGHT_API_LLM_API_KEY` | base64解码 | 硅基流动的 API Key，存在脚本里 |

### 四、恢复指南（出问题时按顺序试）

**场景A：L2 挂了，但 L1 和 L3 还在**

症状：访问 localhost:9177 没反应

恢复：
```bash
# 第一步：看看进程在不在
ps aux | grep hindsight | grep -v grep

# 第二步：如果不在，直接跑启动脚本
bash ~/桌面/restart_hindsight_l2.sh

# 第三步：验证
curl localhost:9177/health
# 应该返回 {"status":"healthy","database":"connected"}

# 第四步：检查记忆还在不在
curl localhost:9177/v1/default/banks/hermes/stats
# 如果 total_nodes > 0，说明数据还在，恢复完成
```

**场景B：L2 的 bank 损坏了（节点没了）**

症状：/health 正常，但 stats 显示 total_nodes=0

恢复：
```bash
# 第一步：删掉损坏的 bank
curl -X DELETE http://localhost:9177/v1/default/banks/hermes

# 第二步：重建空 bank
curl -X PUT http://localhost:9177/v1/default/banks/hermes \
  -H "Content-Type: application/json" \
  -d '{"name":"hermes","disposition":{"skepticism":3,"literalism":3,"empathy":3}}'

# 第三步：从 L1（内部记忆）和 L3（本文件）重建重要事实
# L1 的路牌在 memory tool 里，L3 的信息就在本文件👇
```

**场景C：L1 也丢了（全部失忆）**

灾难恢复顺序：
1. 启动 L2：`bash ~/桌面/restart_hindsight_l2.sh`
2. 从 L2 用 recall 搜索，看能不能找回记忆
3. 从 L3（本文件夹的 MD 文件）手动读取配置
4. 找小龙虾帮忙重建 L1 内部记忆

### 五、验证 L2 是否正常工作的测试

```bash
# 1. 健康检查
curl http://localhost:9177/health
# 预期: {"status":"healthy","database":"connected"}

# 2. 存储统计
curl http://localhost:9177/v1/default/banks/hermes/stats | \
  python3 -c "import json,sys;d=json.load(sys.stdin);print(f'节点:{d[\"total_nodes\"]} 文档:{d[\"total_documents\"]} 观察:{d[\"total_observations\"]}')"

# 3. 语义检索（看能不能找到记忆）
curl -X POST http://localhost:9177/v1/default/banks/hermes/memories/recall \
  -H "Content-Type: application/json" \
  -d '{"query":"三层记忆架构是什么？","budget":"mid"}' | \
  python3 -c "import json,sys;d=json.load(sys.stdin);[print(f'→ {r[\"text\"][:100]}') for r in d.get('results',[])]"

# 4. 存储一条新事实（测试写入）
curl -X POST http://localhost:9177/v1/default/banks/hermes/memories \
  -H "Content-Type: application/json" \
  -d '{"items":[{"content":"这是恢复后的测试写入","context":"恢复测试"}]}'
```

### 六、已知坑（搞之前必读）

1. **HF_HUB_OFFLINE=1 必须设**，不然启动时 sentence-transformers 会尝试联网下载模型，国内网络连不上 HuggingFace，会卡死
2. **max_tokens 不能超 32768**，Qwen2.5-7B 的上下文限制
3. **Reranker 模型 cross-encoder 已缓存**，但如果清理了 HF 缓存需要重新下载
4. **PostgreSQL 是内嵌的**，不需要手动启动，由 hindsight-api 自动管理
5. **端口 9177** 不要被其他进程占用
6. **内存占用约 1GB**（含嵌入模型+PostgreSQL），机器剩余 3.5GB 完全够

## L2向量库使用工作流（2026-05-16 Hermes+Qwen-Max联合设计）

### L2存在意义
花1.2GB内存(固定)换每天省检索token(持续性)。让L1能瘦下来——2000字只存铁律+路牌，细节放L2随取随用。完全解决L1 2000字硬限制问题。

### 检索路径（铁律）
```
L1无答案 → 先搜L2(hindsight_recall) → 再读L3(read_file) → 再翻历史(session_search)
```

### 触发条件
| 条件 | 举例 | 行动 |
|:----|:----|:----|
| L1有"→L2可查"标记 | "【持仓→L2可查】" | 必搜L2 |
| 不确定但感觉存过 | 用户问配置/工具/偏好相关 | 搜L2确认 |
| 概念性问题 | "三层记忆的原理" | 搜L2语义匹配 |
| 事实性问题 | "芯片ETF代码" | 直接L3 read_file更准 |

### 检索技巧
- 用**核心关键词**搜L2，不要用整句用户原话
- 搜不到→换关键词重试一次→再失败才走L3
- hindsight_recall(query)用于常规语义搜索
- hindsight_reflect(query)用于多条相关记忆需合成回答的场景

### 该存L2 vs 该存L3
| 存L2 | 存L3 |
|:----|:-----|
| 用户偏好/习惯 | 完整投资分析文档 |
| API配置和工具用法 | 环境配置指南 |
| 投资分析框架要点 | 灾难恢复手册 |
| 长期观察事项 | 项目计划/待办 |
| 架构审核结论 | 博主观点追踪表 |
| 一句话可说完的知识 | 长篇需要read_file的文档 |
