## ② 拉视频列表（转录即天然限速器）

### 🔄 核心规则：转录就是天然限速器

**不需要sleep，不需要刻意三平台交替。** 视频下载后丢进容器转录，本身就花1-2分钟。这段时间天然就是API冷却期，比sleep(12秒)长10倍。按顺序一个人一个人跑，转录时间自动帮你限速。

```text
正确流程（转录即限速）：
  抖音付鹏 API → 发现新视频 → 下载 → 转录（1-2分钟，天然冷却）
  ↑ 转录完，冷却时间早已过去，安全
  抖音蒋宇飞 API → 发现新视频 → 下载 → 转录（1-2分钟）
  ↑
  B站宋鸿兵 API → 发现新视频 → 下载 → 转录（1-2分钟）
  ↑
  知乎环中星鉴 API → 直接读文字（秒级，知乎限流宽松，无所谓）
  ↑
  好处：不用sleep，不用交替，转录本身就是限速器

错误做法（只有0新时才危险）：
  如果某个博主0新（不用转录），直接跳到下一个人
  → 连续N个0新 = 连续N次API无间隔 = 可能触发限流
  → 解决：0新时sleep(12秒)再请求下一个。有转录时不需要sleep。
```

**⚠️ 请求频率铁律（违反必-799/403）：**
- **有新视频要转录** → 转录时间（1-2分钟）天然限速，不用sleep
- **0新不用转录** → 必须sleep(12秒)再请求下一个，否则连续API会被限
- **知乎纯文字** → 限流宽松，秒级完成也问题不大，但保险起见也sleep(5秒)

## ⑥ 下载+转录

**⚠️ 本阶段核心规则：转录就是天然限速器。** 阶段一发现新视频后，下载→送容器→转录→等完成→取回文本→再处理下一个博主。转录花1-2分钟，这段时间天然就是API冷却期。

**⚠️ 核心规则（违反必卡住）：**
1. **ffmpeg在容器里，主机不需要装！** 直接把视频送容器，脚本自己提音频
2. **`transcribe_new.py` 用位置参数**，不是 `--input/--output`：`python3 /app/temp/transcribe_new.py /app/temp/video.mp4 /app/temp/result`
   - 它在容器内自动提音频→转文字→生成 `result_全文.txt`
3. **卡住了先读脚本源码**，不要自己猜参数！`docker exec videocaptioner head -30 /app/temp/transcribe_new.py`
4. **下载视频不要传cookie**，只传 `-H "Referer: https://www.douyin.com/"`

```bash
# 典型流程（完整可执行版）——转录完再处理下一个：
# 1. 找play_url（从API返回json的 aweme_list[].video.play_addr.url_list[0]）
# 2. 下载视频
curl -L -o /tmp/v_博主名.mp4 -H "Referer: https://www.douyin.com/" "{play_url}"
# 3. 送容器
sg docker -c "docker cp /tmp/v_博主名.mp4 videocaptioner:/app/temp/v_博主名.mp4"
# 4. 容器内转录（自动提音频+转文字）—— 等完成！
sg docker -c "docker exec videocaptioner python3 /app/temp/transcribe_new.py /app/temp/v_博主名.mp4 /app/temp/result_博主名"
# → 等转录完成（1-2分钟），这段时间=天然API冷却
# 5. 取回转录文本
sg docker -c "docker cp videocaptioner:/app/temp/result_博主名_全文.txt /tmp/transcript_博主名.txt"
# 6. 清理容器内外临时文件
sg docker -c "docker exec videocaptioner rm -f /app/temp/v_博主名.mp4 /app/temp/result_博主名.srt /app/temp/result_博主名_全文.txt"
rm -f /tmp/v_博主名.mp4
# 7. 读转录文本 → 进入阶段三分析
cat /tmp/transcript_博主名.txt
# → 转录完，冷却时间早已过去，安全请求下一个博主的API
```

大视频（10分钟+）约2分钟完成。知乎文章直接读文字，不用转录。
