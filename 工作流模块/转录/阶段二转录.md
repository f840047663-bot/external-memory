# 阶段二转录（视频→容器→mp3→文字）+ 重试失败项

## 输入
| 参数 | 类型 | 说明 |
|:----|:----|:----|
| 待转录列表 | 列表 | 来自阶段一的新视频路径 |
| 失败项列表 | 列表 | 来自/tmp/monitor_status.md中❌的项 |

## 输出（返回值）
```
{
  "转录结果": [{"视频ID": "xxx", "状态": "✅/❌", "文本路径": "xxx"}],
  "重试结果": [{"博主": "xxx", "状态": "✅/❌", "原因": "xxx"}],
  "最终待分析列表": ["所有转录完成+知乎文本的路径"]
}
```

## 执行步骤

### Step 1：检查VideoCaptioner容器
```bash
docker ps --filter name=videocaptioner
# 未运行 → docker start videocaptioner
```

### Step 2：转录视频（逐个处理）
**⚠️ 核心规则：转录就是天然限速器。** 转录完一个再处理下一个。

```bash
# 1. 送视频到容器
sg docker -c "docker cp ~/.hermes/work/videos/{博主}_{ID}.mp4 videocaptioner:/app/temp/v.mp4"

# 2. 提mp3（必须！transcribe_new.py只接受音频文件）
sg docker -c "docker exec videocaptioner ffmpeg -i /app/temp/v.mp4 -vn -acodec mp3 -y /app/temp/v.mp3"

# 3. 转录——等完成！
sg docker -c "docker exec videocaptioner python3 /app/temp/transcribe_new.py /app/temp/v.mp3 /app/temp/result"

# 4. 取回文字
sg docker -c "docker cp videocaptioner:/app/temp/result_全文.txt ~/.hermes/work/transcripts/{博主}_{ID}.txt"

# 5. 清理容器临时文件
sg docker -c "docker exec videocaptioner rm -f /app/temp/v.mp4 /app/temp/v.mp3 /app/temp/result.srt /app/temp/result_全文.txt"
rm -f ~/.hermes/work/videos/{博主}_{ID}.mp4

# 6. 更新状态清单
echo "| 转录 | {博主} | 视频 | ✅ | - | transcripts/{博主}_{ID}.txt | 待分析 |" >> /tmp/monitor_status.md
```

**⚠️ 踩坑记录：**
- `transcribe_new.py`只接受mp3/wav，**不接受mp4！** 必须先ffmpeg提音频
- 如果报"Unsupported sound format: mp4"→ 必须先ffmpeg提mp3
- 抖音CDN把音视频分成两个独立流→ 如果只有h264无aac→ 需合并（见[通用/Cookie提取]抖音特殊处理）

**🚫 禁止：**
- 直接传mp4给转录脚本
- 装ffmpeg到主机（ffmpeg在容器里，主机不需要）
- 遇到error马上切方案（先读脚本源码：`docker exec videocaptioner head -30 /app/temp/transcribe_new.py`）

### Step 3：重试阶段一的失败项
**🚨 铁律：阶段一标记为"失败-待重试"的项，必须在阶段二回头重试！不能放弃！**

```bash
# 读/tmp/monitor_status.md，找出所有❌的项
# 逐个重试（sleep(15秒)后再请求）
```

```
重试流程：
  失败项 → sleep(15s)重试
    ├─ 成功 → 保存到暂存目录，更新状态清单为✅
    └─ 还失败 → sleep(20s)再试
        ├─ 成功 → ✅
        └─ 还失败 → 检查cookie
            ├─ cookie无效 → 用cdp-get-cookies.py刷新 → 再试
            └─ cookie有效还失败 → 标记"最终失败"，汇报用户
```

### Step 4：输出最终状态
```
✅ 最终成功：X条
❌ 最终失败：Y条（原因：...）→ 需要用户回来处理
```

## 任务编排原则
```
能立即处理的 → 先处理（转录/分析已就绪的内容）
需要浏览器的 → 排到"浏览器队列"，全部API任务跑完后再集中处理
需要刷新cookie的 → 标记"待刷新"，最后统一刷新
需要用户操作的 → 标记"待用户"，全部自动任务跑完后一次性汇报
```

**🚫 禁止：** 遇到一个障碍就停下来等用户，把其他能做的任务都搁置。
**✅ 正确：** 能做的全做了，最后汇总所有卡住的点，一次性汇报。
