# 模型路由与成本优化方案分析

> 写于：2026-05-15
> 目的：全面分析降低日常对话token成本的各种方案，供大好人决策
> 背景：当前默认模型DeepSeek V4 Flash（约¥50-60/月），日常闲聊也用它在跑，浪费严重

---

## 一、当前token消耗画像

### 每轮完整会话的token构成

| 组成部分 | 预估token | 说明 |
|:---------|:----------|:-----|
| 系统提示词核心指令 | ~2,000 | 角色定义、工具使用规则等 |
| 内部记忆（2,200字中文） | ~1,500 | 持仓/铁律/框架，中文token效率低 |
| 用户画像（650字） | ~400 | 用户偏好、Ubuntu水平等 |
| 可用技能列表 | ~500 | 数十个技能名称+描述 |
| 工具定义（全部20+个工具schema） | ~3,500 | 每个工具的JSON Schema |
| 当前上下文（session信息等） | ~500 | 平台、目录等 |
| **每轮固定开销** | **~8,500** | **不管你问什么，这8.5K先烧了** |

> 问一句"吃了没" = 烧8.5K token的系统提示词成本
> DeepSeek V4 Flash约¥0.5/1M tokens，一次闲聊约¥0.0043
> 看起来单次便宜，但每天几十条闲聊累积起来就是大头

### 可优化的部分

| 部分 | 能否去掉 | 节省token |
|:----|:---------|:----------|
| 核心系统指令 | ❌ 不能 | 0 |
| 内部记忆+用户画像 | ✅ 可以 | ~1,900 |
| 技能列表 | ✅ 可以 | ~500 |
| 工具定义 | ⚠️ 可精简 | ~2,000+（只留消息发送工具） |
| 当前上下文 | ⚠️ 部分可去 | ~300 |

## 二、方案一：轻量Profile（推荐首选）

### 思路
创建独立的Hermes profile，系统提示词极度精简：
- 无记忆注入（不加载内部记忆+用户画像）
- 无技能列表
- 只保留最低限度工具（仅`send_message`）
- 搭配阿里云Qwen或硅基流动Qwen2.5-7B

### 精简后的profile系统提示词
```
系统提示词：~2,000（核心指令不可省）
工具定义：~500（只保留1-2个消息工具）
总计：~2,500 token
```
对比当前 ~8,500 token，**节省 ~70%**

### 与当前会话的关系

```
主会话（DeepSeek V4 Flash）       轻量会话（阿里云Qwen/硅基流动Qwen）
┌─────────────────────┐          ┌──────────────────────┐
│ 完整系统提示 ~8.5K  │          │ 精简系统提示 ~2.5K   │
│ 工具齐全             │  ───→   │ 仅消息发送工具        │
│ 复杂任务专用         │  relay  │ 日常闲聊专用          │
│ 投资分析/写代码/看图 │  ←───   │ 回复内容传回来        │
└─────────────────────┘          └──────────────────────┘
```

### 工作方式
1. **我判断任务类型**：简单闲聊 vs 复杂任务
2. **简单闲聊** → 我调便宜模型API，只传精简提示+用户消息，拿到回复后念给你
3. **复杂任务** → 正常走DeepSeek工具链

### 优势
- ✅ 架构清晰，不会搞坏主配置
- ✅ 每次闲聊省 ~6,000 token（~70%）
- ✅ 阿里云qwen-turbo免费额度 ≈ ¥0/月
- ✅ 硅基流动Qwen2.5-7B ≈ 你已有免费额度
- ✅ 紧急时可切回DeepSeek兜底

### 劣势
- ⚠️ 需要我"手动判断"任务类型，判断错了会走错模型
- ⚠️ 轻量模型无工具调用能力（不能执行命令、读写文件）
- ⚠️ relay模式有额外网络延迟（~0.5-2秒）
- ⚠️ 实现需要写一个转发脚本 + 测试稳定

### 预计实现时间
- 写转发脚本：15分钟
- 测试阿里云Qwen/硅基流动两路：10分钟
- 纳入日常使用：立即可用

## 三、方案二：Script Relay（最低侵入方案）

### 思路
不创建新profile，而是在当前会话内通过终端调用便宜模型API。

### 工作方式
```
用户问简单问题
  → 我识别为"简单闲聊"
  → terminal() 调用脚本: curl 便宜模型API, 传精简提示+用户问题
  → 拿到回复, 直接念给用户
  → 全程不走DeepSeek
```

### 优势
- ✅ 不需要改任何Hermes配置（零风险）
- ✅ 不需要新会话、不用切模型
- ✅ 当前会话内就能用

### 劣势
- ⚠️ terminal()有额外开销（每次调用脚本也要token）
- ⚠️ 还是需要我手动判断复杂度
- ⚠️ 比较hacky，不够优雅

### 与方案一的本质区别
方案一=建双车道（两个profile）
方案二=在当前车道上临时开个小车（curl API）

## 四、方案三：本地模型

### 硬件评估

当前机器：**Intel i5-3340M (2012年)** | **无NVIDIA GPU** | **7.1GB RAM**

| 模型 | 量化 | 内存需求 | 推理速度（纯CPU） | 质量评估 |
|:----|:----|:---------|:-----------------|:---------|
| Qwen2.5-0.5B | Q4 | ~0.4GB | ~20 tok/s | ❌ 几乎不能用 |
| Qwen2.5-1.5B | Q4 | ~1GB | ~10 tok/s | ⚠️ 勉强能聊天，常答非所问 |
| Qwen2.5-3B | Q4 | ~2GB | ~5 tok/s | ⚠️ 简单对话可以，但不太聪明 |
| Qwen2.5-7B | Q4 | ~5GB | ~1-2 tok/s | ✅ 能聊天，但慢到无法忍受 |

### 推荐评估

**在现有硬件上，唯一可能的是Qwen2.5-3B Q4（Ollama）**：
- 安装Ollama：10分钟
- 下载模型：20-40分钟（2GB，取决于网速）
- 配置Hermes调用本地API：10分钟
- 总时间：**~1小时**
- 效果：能简单聊天，但回复质量明显低于API模型，且速度较慢

**如果换机器（用户曾提过）**：
- 最低建议：二手NVIDIA GTX 1050 (~¥300) → 可流畅跑7B Q4
- 中等建议：RTX 3060 12GB (~¥1,500-2,000) → 可跑13B-30B
- 理想建议：RTX 4090 → 可跑本地70B

### 劣势
- ⚠️ 当前硬件太老，本地模型体验差
- ⚠️ 磁盘仅剩37G，下载模型空间紧张
- ⚠️ 每秒1-5个token，聊天体验卡顿
- ⚠️ O(1小时)配置时间，且可能踩坑

### 结论
**当前硬件不建议上本地模型**。体验差、效果差、性价比不如API方案。
如果未来换带NVIDIA显卡的机器，再重估。

## 五、方案四：多API路由

### 现状

```yaml
# 当前config.yaml已配置
providers:
  deepseek:  ✅ 主力，¥46余额
  dashscope: ✅ 阿里云Qwen，¥5余额
  # siliconflow: ❌ 有Key但未写入，等大好人给条件
```

### Hermes原生路由能力

| 能力 | 支持情况 |
|:----|:---------|
| 自动按复杂度路由 | ❌ 完全不支持 |
| Provider fallback（A挂了切B） | ❌ `fallback_providers: []` 空，需手动 |
| Credential pool（多Key轮换） | ✅ 支持，但仅同一provider多Key |
| Profile多配置切换 | ✅ 支持，需新会话生效 |
| /model 命令切换 | ✅ 支持，需 /reset 新会话 |

### 可行的多API部署方式

#### 方式A：Credential Pool（多Key轮换）
- 同一provider下多个API key
- DeepSeek场景：可注册多个DeepSeek账号，把Key加入池
- 对当前场景无帮助（我们要的不是同provider轮换，是不同model按场景路由）

#### 方式B：子代理路由（delegate_task）
```
主会话（便宜模型）        子代理（贵模型）
┌──────────────┐        ┌──────────────┐
│ 日常闲聊     │ ───→   │ 复杂任务     │
│ Qwen2.5-7B   │        │ DeepSeek V4  │
│ 无工具       │ ←──── │ 全工具       │
└──────────────┘        └──────────────┘
```
- 需要把delegation.model配置为DeepSeek
- 主会话走便宜模型（无工具能力，仅聊天）
- 复杂任务委托给子代理（全工具+DeepSeek）
- **问题是：所有消息先经过便宜模型判断"是简单还是复杂"**，这个判断质量存疑

#### 方式C：手动双Profile
```
hermes --profile cheap     → 轻量闲聊
hermes --profile default   → 复杂任务
```
每次切换开新会话。最稳但最麻烦。

## 六、方案综合对比

| 维度 | 轻量Profile (推荐) | Script Relay | 本地模型 | 多API路由 |
|:----|:------------------:|:------------:|:--------:|:---------:|
| **token节省** | ~70% | ~70% | 100%（免费） | ~50-70% |
| **实现时间** | ~25分钟 | ~15分钟 | ~1小时+ | ~15分钟 |
| **配置风险** | 低（新建profile） | 极低（不改配置） | 中（装Ollama） | 低 |
| **稳定性** | 高 | 中（hacky） | 低（硬件差） | 高 |
| **对话体验** | 好（秒回） | 好（有relay延迟） | 差（1-5 tok/s） | 好 |
| **硬件要求** | 无 | 无 | 高（当前不满足） | 无 |
| **维护成本** | 低 | 低 | 中（要关注进程） | 低 |
| **扩展性** | 好（可加更多profile） | 差（纯hack） | 好（换好卡后） | 好 |

## 七、推荐方案（混合部署）

### 第一优先级：轻量Profile + Script Relay
```
日常对话走什么模型，由我判断（大好人不用管）：
┌────────────────────────────────────────────┐
│  用户发消息                                  │
│      ↓                                       │
│  Hermes判断任务类型                          │
│   ├─ 简单闲聊/问候/确认 → 调便宜模型API     │
│   │   (阿里云qwen-turbo 或 硅基流动Qwen7B)   │
│   │   精简提示词 ~500 token                  │
│   │                                          │
│   └─ 复杂/投资分析/代码/看图 → DeepSeek V4  │
│      完整提示词 ~8,500 token + 全工具        │
│                                              │
│  用户完全无感，只知道回复回来了               │
└──────────────────────────────────────────────┘
```

### 第二优先级：加硅基流动
等大好人给出具体条件后，把硅基流动Key写入`.env`，作为第三路provider

### 第三优先级：本地模型（换好机器再考虑）
当前硬件（i5-3340M + 7G RAM）不适合。换带NVIDIA显卡的机器后，可以用Ollama跑7B Q4，日常闲聊0成本

### 不推荐的做法
- ❌ 全局切阿里云qwen-turbo → 上次已验证，无工具调用 = 系统瘫
- ❌ 本地模型硬跑 → 效果差+速度慢+空间紧
- ❌ 强行部署子代理路由 → 增加复杂度，判断质量不可控

## 八、最终决策（2026-05-15 大好人确认）

```
日常闲聊 → 硅基流动 Qwen2.5-7B-Instruct（免费）
复杂任务 → DeepSeek V4 Flash（主力）
阿里云百炼 → 有付费额度，备用
```

### 实现方式：Script Relay

脚本路径：`~/.hermes/scripts/chat-relay.sh`

用法：我判断是闲聊时，自动调这个脚本调用硅基流动API，把回复转发给你。
用户无感知，相当于我在DeepSeek上干活，闲聊借硅基流动的免费额度。
