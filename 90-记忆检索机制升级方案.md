## 一、来龙去脉

### 起因

2026-06-13讨论经济危机观点汇总时，用户问"你是咋拉取的"，我说`grep -rl "关键词" events/*.md`。用户惊讶发现Linux自带grep有这个能力，而ta平时用Windows不知道。由此引发思考：

> **用户原话：**「那实际上这不是跟向量库差不多吗？那实际上你可以用这个命令去，把我那么多md文件也关联起来呀」

### 当前架构

```
记忆检索三层（实际运行中）：
L1（内部记忆 2200字）— 铁律+路牌，每轮自动注入
L2（Hindsight向量库）— 端口9177，PostgreSQL+Embedding模型
L3（external_memory/）— events/*.md + 各类知识图谱
```

### 发现的问题

1. **L2向量库太脆弱**：71个僵尸进程、1.7GB内存、OOM被杀手
2. **用户的核心需求其实就是关键词搜索**：问的问题（"经济危机""AI泡沫"）关键词明确，归档时已经标好了核心词
3. **grep零成本零风险**：原生Linux命令，永远不会崩

### 为什么现在不改

> **用户原话：**「你仔细研究一下，有啥向量库比他好的地方，然后他比向量库好的地方」「不要轻举万动，这个向量库用了这么久了，没必要就着急忙慌的把它改过来」

当前策略：**L2向量库先继续用（已经重启好了），哪天它又崩了再切方案。到时候方案已经想清楚了。**

## 二、三种方案的对比

### 方案A：现状——L2向量库（Hindsight）

| 维度 | 评分 | 说明 |
|:----|:----:|:-----|
| 内存占用 | 🔴 1.7GB | 含Embedding模型+PostgreSQL |
| 可靠性 | 🔴 天天崩 | 僵尸进程堆叠、OOM、端口被占 |
| 语义搜索 | 🟢 强 | 同义词/近义词都能找到 |
| 关键词搜索 | 🟢 强 | 但不如grep精确 |
| 维护成本 | 🔴 高 | 要养daemon+PostgreSQL+定期清理 |
| 速度 | 🟡 几百ms | 有网络延迟（走本地API） |

**最佳场景：** 记不清关键词，只说"那个谁说市场要崩的"这种模糊回忆

### 方案B：grep（原生Linux命令）

| 维度 | 评分 | 说明 |
|:----|:----:|:-----|
| 内存占用 | 🟢 0 | 纯命令，用完即走 |
| 可靠性 | 🟢 永远不崩 | Linux内核自带 |
| 语义搜索 | 🔴 无 | 只能精确匹配关键词 |
| 关键词搜索 | 🟢 极强 | grep -rl "危机" 一秒出结果 |
| 维护成本 | 🟢 0 | 不需要任何维护 |
| 速度 | 🟢 几十ms | 纯文件I/O |

**最佳场景：** 关键词明确（"AI泡沫""宋鸿兵""2027"），且归档时已标好核心词

### 方案C：sqlite3 FTS5（推荐升级路径①）

| 维度 | 评分 | 说明 |
|:----|:----:|:-----|
| 内存占用 | 🟢 极小 | 仅索引文件，无daemon |
| 可靠性 | 🟢 高 | sqlite3极其稳定 |
| 语义搜索 | 🟡 中等 | 支持分词+词干匹配，但不是真语义 |
| 关键词搜索 | 🟢 极强 | 比grep更强（可排序、可权重） |
| 维护成本 | 🟢 低 | 一句Python脚本重建索引 |
| 速度 | 🟢 极快 | 索引查询 |

**最大优势：** 零依赖、零安装（Python标准库自带sqlite3+s，FTS5已编译进系统sqlite3）。比grep强的点是：**匹配密度排序**（搜"危机"→匹配次数多的文件排前面），支持**中文分词**。

### 方案D：本地Embedding（推荐升级路径②）

| 维度 | 评分 | 说明 |
|:----|:----:|:-----|
| 内存占用 | 🟡 几十~几百MB | only-my加载模型时 |
| 可靠性 | 🟢 中上 | 没有daemon，但进程内算 |
| 语义搜索 | 🟢 强 | 同向量库一样 |
| 关键词搜索 | 🟢 强 | 可用grep兜底 |
| 维护成本 | 🟡 中 | pip install sentence-transformers |
| 速度 | 🟡 几秒 | 首次加载模型慢 |

**最佳场景：** 需要真语义搜索但不想养PostgreSQL daemon。每次Hermes启动时加载模型，进程内算相似度，用完即走。

### 方案E：sqlite-vec（推荐升级路径③——融合方案）

| 维度 | 评分 | 说明 |
|:----|:----:|:-----|
| 内存占用 | 🟢 极小 | sqlite3扩展，无独立进程 |
| 可靠性 | 🟢 高 | sqlite3基金会维护 |
| 语义搜索 | 🟢 强 | 直接在sqlite3里跑向量搜索 |
| 关键词搜索 | 🟢 极强 | FTS5+vec可共存，一张表同时搜 |
| 维护成本 | 🟡 中 | 需要装扩展（pip install sqlite-vec） |
| 速度 | 🟢 快 | 本地索引 |

**最大优势：** FTS5（关键词）和vector（语义）两个引擎共存，一张表里同时搜两种，极端轻量。

## 四、升级路径规划

### 当前（阶段0）：向量库照常运行
L2向量库已经重启成功，正常工作。不主动切换。

### 下次向量库崩了（阶段1）：切grep兜底
```
cd ~/.hermes/external_memory/events
grep -rl "关键词" *.md | xargs -I{} sh -c 'echo "=== {} ==="; grep -H "关键词" {}'
```
零安装、零依赖，3分钟切完。

### 用户说"搞一下"（阶段2）：上sqlite3 FTS5
原因：比grep多排优先级+中文分词，比向量库轻100倍，且零安装（Python自带）。

**实施步骤：**
```python
import sqlite3, os, re, glob
from datetime import datetime

# 1. 建库建索引
conn = sqlite3.connect('/tmp/events_fts.db')
conn.execute('CREATE VIRTUAL TABLE events USING fts5('
    'title, content, date, source, tokenize=porter unicode61)')

# 2. 导入events目录所有.md文件
events_dir = os.path.expanduser('~/.hermes/external_memory/events/*.md')
for fpath in glob.glob(events_dir):
    fname = os.path.basename(fpath)
    with open(fpath) as f:
        content = f.read()
    conn.execute("INSERT INTO events(title, content, date, source) VALUES (?,?,?,?)",
        (fname, content, '', ''))

# 3. 搜索
for row in conn.execute(
    "SELECT title, rank FROM events WHERE events MATCH ? ORDER BY rank LIMIT 10",
    ('危机',)):
    print(row)
```

全部代码不超过50行，跑完即用，不需要daemon。可以封装成 `~/桌面/events-search.sh` 方便终端直接调用。

### 如果后期需要真语义（阶段3）：在FTS5基础上叠Embedding
搜索时先FTS5出前50条，再Embedding rerank。两不耽误。
