# 监控统筹框架

> **最后更新：** 2026-05-31
> **设计原则：** 状态驱动，不是判断驱动。AI不需要"想怎么走"，只需要"读状态→执行→写状态"。

---

## 执行顺序（从上到下，不要跳）

```
[通用/环境准备] → [抓取/阶段一] → [转录/阶段二] → [分析/阶段三] → [收尾/阶段四]
                                                              ↓
                                                    有API失败项且用户同意？
                                                              ↓ 是
                                                     [浏览器/兜底]
                                                              ↓
                                                    回到对应阶段重试
```

## 状态文件：`/tmp/monitor_state.json`

每个模块执行完**必须写**这个文件。下一个模块**必须读**这个文件。

```json
{
  "当前Step": "环境准备完成",
  "Cookie": {"抖音": "✅", "B站": "✅", "知乎": "❌"},
  "开关": {"调试": "要", "宏观": "不要", "待重仓": "不要", "三关": "不要"},
  "抓取结果": {},
  "失败项": [],
  "待转录": [],
  "待分析": [],
  "转录结果": [],
  "贝叶斯结果": [],
  "断点已更新": false
}
```

## 执行规则

1. **按顺序执行**，不要跳步
2. **每步之间停顿**，输出结果，等用户说"继续"
3. **阶段一有失败项** → 问用户"是否用浏览器兜底？" → 用户说要 → 调[浏览器/兜底] → 回到阶段一重试
4. **全部完成** → 强制自检10项 → 汇报用户
5. **中断恢复** → 读状态文件，从断点Step继续

## 强制自检清单（阶段四执行，缺任何一条不准发消息）

```
□ ①主页ID从INDEX.md读的  ②长期主题≥7天搜了  ③P值链可追溯
□ ④P'虚值标了  ⑤events写了  ⑥hindsight_retain存了
□ ⑦状态清单写了  ⑧失败项重试了  ⑨内容存暂存目录  ⑩三平台交替了
```
