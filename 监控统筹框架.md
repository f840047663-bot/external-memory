## 🔑 断点日期（最重要的变量）

**断点日期 = Excel看板中最后一条有P值更新内容的日期。**

### 确定方法
1. 打开 `~/桌面/持仓贝叶斯跟踪.xlsx`
2. 看 **Sheet2**（贝叶斯完整记录）中最后有内容的行的日期 → 这就是断点日期
3. 如果Sheet2为空或只有表头 → 看 **总看板** 中最后有内容的行的日期
4. 断点日期确定后，**抓取起点 = 断点日期 + 1天**

### 为什么看Excel不看断点文件？
- 断点文件是人写的，可能遗漏/写错
- Excel看板是最终交付物，有P值更新=真正走完管道
- **断点文件 = 看板的镜像**，看板是源，断点文件是副本
- 看板里有啥，断点就记到啥；看板清空了，断点也要回退

### 示例
```
Sheet2最后有内容的行: 2026-06-01 → 断点日期=2026-06-01 → 从2026-06-02开始抓
Sheet2最后有内容的行: 2026-05-30 → 断点日期=2026-05-30 → 从2026-05-31开始抓
```

### ⛔ 铁律
- **必须先确定断点日期，才能开始抓取。** 不准跳过这步。
- 断点日期不确定 → 汇报用户，不准猜。

### 抓取执行方式
- **用background terminal跑脚本**：`python3 ~/.hermes/scripts/scrape_all_platforms.py`
- 脚本自动从Excel读断点日期，三平台交替抓取
- **不准用子agent抓取**（会被用户消息中断）
- 脚本输出日志到 `~/桌面/监控运行日志.md`，JSON结果到 `/tmp/scrape_full_output.txt`

## 核心概念：断点 vs 日志 vs 看板

| 概念 | 文件 | 功能 | 更新时机 | 断了怎么办 |
|:----|:----|:----|:----|:----|
| **断点** | `~/桌面/监控处理断点.md` | 记录每个博主**最后完整走完管道的视频ID+时间** | 阶段四，管道**全部走完**才更新 | 不更新，下次重新捕获（浪费token但不丢信息） |
| **日志** | `~/桌面/监控运行日志.md` | 逐条记录每条视频的**处理过程**。审计痕迹，不清空 | 每处理完一条就append一行 | 读最后20行恢复 |
| **看板** | `~/桌面/持仓贝叶斯跟踪.xlsx` | **最终输出**，用户看的。事件链+P值+结论 | 阶段四一次性批量写入 | 不准阶段三逐条写 |

**断点=看板镜像。** 看板里有啥，断点就记到啥。看板清空了，断点也要回退。

**断点更新铁律：断点更新和Excel看板更新绑定，必须同时发生。**
- Excel没更新 → 断点不准动
- 断点动了 → Excel必须已经同步
- 两者不一致 = 断档事故

**关键规则：**
1. 断点=管道全部走完+Excel写完才能更新，断了不更新（浪费token但不丢信息）
2. 看板只在阶段四一次性批量写入，避免中断污染
3. 日志不清空，新任务追加新日期标题

## §决策树与失败处理

> **遇到问题先走这里，不要自己猜。** 这是最高频读取的内容。

### 第1步：API返回空/JSON解析失败
```
第1次→空/JSON失败 → sleep(10s) → 第2次→成功✅
                             →失败→sleep(15s)→第3次→成功✅
                                   →失败→检查Cookie(wc -c)
                                     ├─ <500B→调§0刷新→再试
                                     └─ >1000B→标记"失败-待重试"，继续下一个
```
**知乎特别注意：** 返回空**不是cookie过期**，第一反应是sleep→重试！

### 第2步：API返回-799/403/限流
```
收到-799/403 → sleep(15s) → 重试1次→成功✅
                   →还限流→换下一个平台（三平台交替）
                   →阶段二回头重试（冷却时间够了大概率成功）
```
**🚫 禁止：** 连续重试同一平台 / 限流了只sleep(5秒)

### 第3步：转录报"Unsupported sound format: mp4"
原因：transcribe_new.py只接受mp3/wav。解决：先ffmpeg提mp3（详见§阶段二）。

### 第4步：抖音视频只有h264无aac音频
抖音CDN音视频分流。解决：调`工作流模块/浏览器/兜底.md`→Network拦截两个URL→ffprobe区分→ffmpeg合并。

### 第5步：以上都试了还不行
```
汇报格式：❌ 失败项：{平台}-{博主}-{类型}
已尝试：1.sleep(10s)→失败 2.sleep(15s)→失败 3.Cookie检查→{有效/已刷新}
需要你回来处理。
```

### 🚫 禁止行为
```
❌ 空返回就跳过（必须sleep重试至少2次）
❌ 空返回就刷新Cookie（可能是间歇性问题）
❌ 连续重试同一平台不换气
❌ 遇到error自己猜方案
❌ Cookie无效还继续跑
❌ 转录直接传mp4
```

### 排错速查表

| # | 问题 | 修复 |
|:--|:----|:----|
| 1 | 抖音count太小→空列表 | 永远用count=30 |
| 2 | 抖音UA太笼统→空响应 | 用完整Chrome版本号UA |
| 3 | 抖音Referer不带user路径 | Referer精确到/user/{sec_uid} |
| 4 | 抖音API非时间倒序 | 先搜断点ID位置，再比ID大小 |
| 5 | 抖音ID不严格单调 | ID+时间双验证 |
| 6 | JSON解析失败≠API挂了 | 子进程timeout设120秒+ |
| 7 | 知乎文章vs回答 | 拆两行各自独立追踪 |
| 8 | sleep不足→限流 | sleep(12秒)，不着急 |

## §阶段三：分析（V4 Pro优化版）

> **核心改动：** 从"一次性读所有转录"拆解为4个子步骤，逐步收敛。

### 3.1 子步骤一：优先级排序
```
目标：117个转录文件，先读哪些？

排序规则（降序）：
1. 持仓直接相关（标题含芯片/养殖/电池/恒科/银行/有色/稀金）→ 优先级A
2. 宏观/中观沾边（利率/地缘/猪周期/美债）→ 优先级B
3. 博主权重高（付鹏/宋鸿兵/蒋宇飞等核心信源）→ 优先级B
4. 其余 → 优先级C（批量扫描，不逐字读）

执行：
python3 -c "
import glob, os
files = glob('~/.hermes/work/transcripts/*.txt')
keywords_a = ['芯片','养殖','电池','恒科','银行','有色','稀金','AI']
keywords_b = ['利率','地缘','猪周期','美债','去杠杆','霍尔木兹']
core_thinkers = ['付鹏','宋鸿兵','蒋宇飞']
for f in sorted(files, key=os.path.getmtime, reverse=True):
    name = os.path.basename(f)
    if any(k in name for k in keywords_a): print(f'A:{f}')
    elif any(k in name for k in keywords_b) or any(t in name for t in core_thinkers): print(f'B:{f}')
    else: print(f'C:{f}')
"
```

### 3.2 子步骤二：逐文件抽取核心观点
```
对A/B级文件：逐字阅读，提取：
- 关键数据（数字/百分比/日期）
- 逻辑链（因为A所以B所以C）
- 时间节点（何时发生/何时结束）
- 持仓关联度（直接影响哪个持仓）

对C级文件：批量grep关键词，只读命中行
```

### 3.3 子步骤三：跨博主验证（信号强度判定）
```
统计每个观点/信号被多少个博主提及：
- ≥3个博主 → 强信号（红色高亮，P值调整权重×2）
- 2个博主 → 中等信号（黄色高亮）
- 1个博主 → 弱信号（正常权重）

输出格式：
| 信号 | 提及博主数 | 博主列表 | 信号强度 | 建议P值调整 |
|:----|:--------:|:--------|:--------|:----------|
| 养殖右侧布局 | 5 | 付鹏+伊娃+生猪贸易+拿幸+厄尔尼诺 | 🔴强 | +4pp |
| 霍尔木兹封锁 | 3 | 宋鸿兵+闭眼看世界+岩松笔记 | 🔴强 | 待确认 |
```

### 3.4 子步骤四：叙事主线生成（LLM自动化）
```
输入：子步骤三的跨博主验证表 + 子步骤二的核心观点摘要
输出：3条叙事主线（自动生成，不依赖人工总结）

调用方式：
python3 -c "
# 调用V4 Pro API生成叙事主线
import json, urllib.request
prompt = '''基于以下跨博主验证结果，生成3条叙事主线：
{跨博主验证表}
要求：每条主线包含①标题 ②核心逻辑 ③受益/受损资产 ④置信度'''
# POST到V4 Pro API
"
```

**🚨 铁律：**
- 叙事主线必须基于≥2个博主交叉验证，单博主观点不能作为主线
- 叙事主线与事件链必须打通（叙事→传导→具体持仓）
- 叙事主线变动→必须更新看板"叙事分析"Sheet

### 3.5 贝叶斯更新（五道门，缺一不调）
```
A(事件/系统/噪声) → B(宏观折价) → C(机械去偏) → D(定性，读position文件) → E(事实/观点分离+共识衰减+查重)
```
**0pp也必须写理由。**

| 门 | 规则来源 |
|:--|:----|
| A门 | 判断事件/系统性变化/噪声。噪声→±0pp |
| B门 | 利率折价。详见 `00-宏观传导框架.md §利率` |
| C门 | 逻辑簇去重。详见 `21-贝叶斯防偏差规则.md` + `positions/贝叶斯计算标准.md` |
| D门 | 读position文件定性。资产本质变了才大调 |
| E门 | 事实进主P，观点进P'。多源同逻辑只加1~2pp |

### ⛔ P'强制规则
```
未落地事件/预测→不准进主P值，只写虚值P'
P'转实条件：①事件落地 ②价格先行确认(连续3天同向+动因校验) ③超窗口→作废
P'转实=必须推用户确认，不允许自动转实
```

### ⛔ P值链强制校验
```
格式：文件头P = 链累加结果 = 账户确认值
例：养殖ETF 51% = 50-2-2+4+1 = 51% ✅
```

### 4.2.1 🚨 自动生成决策仪表盘（Sheet0）
**每次看板更新后必须执行，不是一次性的！**
