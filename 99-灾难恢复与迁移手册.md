# 🧠 Hermes 记忆迁移与灾难恢复手册

> 最后更新：2026-05-16
> 目的：配置坏了 / 换电脑 / 重装系统时，能无缝恢复全部记忆和能力

---

## 0️⃣ 三层记忆恢复总览（你的保险单）

> **能不能用 MD 文件完全恢复？** 能恢复大部分，但不是全部。下面这张表说明白了。

### ✅ 用 MD 文件能恢复的（纯文本，人人能读）

| 内容 | 位置 | 文件格式 | 占有量 |
|:----|:-----|:--------|:------|
| 🏠 **L1 我的内部记忆** | `~/.hermes/memories/MEMORY.md` | MD纯文本 | ~10KB |
| 🏗️ **Skills（技能）** | `~/.hermes/skills/` | MD纯文本 | ~50KB |
| 📚 **L3 外部知识库** | `~/.hermes/external_memory/*.md`（18个文件） | MD纯文本 | ~100KB |
| ⚙️ **配置文件** | `~/.hermes/config.yaml` | YAML纯文本 | ~5KB |
| 🔑 **API密钥** | `~/.hermes/.env` | 纯文本 | ~1KB |
| 🏃 **定时任务** | `~/.hermes/cron/jobs.json` | JSON | ~3KB |

### ❌ 纯 MD 文件恢复不了的

| 内容 | 为什么 | 怎么办 |
|:----|:------|:------|
| 🧬 **L2 向量库（551条语义记忆）** | PostgreSQL数据库（153MB），不是文本 | **备份 `~/.pg0/` 整个目录**（已加进备份脚本） |
| 💬 **历史对话（session_search）** | SQLite 数据库 `state.db`（2MB） | **备份 `state.db`** |
| 🦞 **小龙虾的记忆/状态** | OpenClaw 自己的数据库 | 小龙虾那边的备份，不是我的事 |

### 极限方案：如果只有 MD 文件，没有数据库备份

> 重装新的 Hermes → 把上述 MD 文件放回去 → 我就能知道你是谁、你持仓有什么、工作流怎么跑、API怎么连。**只是 L2 语义检索能力要从零积累，但核心能力不会丢。**

**恢复后我会丢失的能力：**
- ❌ 记不太清之前讨论过的模糊联想（得重新学）
- ❌ 会话历史全没了（问"我们之前聊过什么"答不上来）
- ✅ 但持仓规则、监控阈值、API配置、定时任务全在
- ✅ 知道你是谁、你有什么持仓、工作流怎么跑
- ✅ 能继续出日报、做跟踪、跟小龙虾协作

**一句话：存好 MD 文件 = 保底活过来。存好 `~/.pg0/` + `state.db` = 跟没坏过一样。**

## 1️⃣ 备份清单

**所有数据都在** `~/.hermes/` 下面，只需备份这一个目录：

| 文件/目录 | 大小 | 重要等级 | 包含内容 |
|-----------|------|---------|---------|
| `~/.hermes/config.yaml` | ~5KB | ⭐⭐⭐ 必须 | API密钥、模型配置、插件设置、MCP配置 |
| `~/.hermes/external_memory/` | ~50KB | ⭐⭐⭐ 必须 | INDEX、第2层+第3层所有MD文件 |
| `~/.hermes/skills/` | ~50KB | ⭐⭐⭐ 必须 | 所有自定义Skills（含daily-briefing等） |
| `~/.hermes/state.db` | ~2MB | ⭐⭐⭐ 必须 | 内部记忆、历史对话、session_search数据 |
| `~/.hermes/cron/jobs.json` | ~3KB | ⭐⭐ 重要 | 所有定时任务配置 |
| `~/.hermes/.env` | ~1KB | ⭐⭐⭐ 必须 | API密钥（DeepSeek/阿里云/OpenRouter） |
| `~/.hermes/processes.json` | ~1KB | ⭐ 可选 | 后台进程状态，丢了可重建 |
| `~/.hermes/kanban.db` | ~10KB | ⭐ 可选 | 任务看板，丢了可重建 |
| `~/.hermes/hermes-agent/` | ~3MB | ⭐⭐ 重要 | Hermes Agent本体代码（如果用git装的可重拉） |
| `~/.pg0/` | **153MB** | ⭐⭐⭐ **必须（L2）** | Hindsight 向量库全部数据（551条语义记忆）⚠️ 在 ~/.hermes/ 外面，单独备份 |

## 2️⃣ 一键备份命令

```bash
# 停服（如果正在运行）
hermes stop  2>/dev/null

# 打包全部数据，文件名带时间戳
# 注意：.pg0/ 在 ~/.hermes/ 外面，单独加进来
tar czf "hermes-backup-$(date +%Y%m%d_%H%M%S).tar.gz" \
  -C ~/ \
  .hermes/config.yaml \
  .hermes/.env \
  .hermes/state.db \
  .hermes/cron/jobs.json \
  .hermes/skills/ \
  .hermes/memories/ \
  .hermes/external_memory/ \
  .pg0/

echo "✅ 备份完成，文件大小：$(ls -lh hermes-backup-*.tar.gz | awk '{print $5}')"
echo "包含：L1记忆+L2向量库(153MB)+L3知识库+配置+Skills+API密钥"
```

**建议：每周末手动跑一次，或设定时任务自动备份。**

## 3️⃣ 在新电脑/重装后恢复

### 第一步：安装 Hermes Agent

```bash
# 跟首次安装一样
git clone https://github.com/NousResearch/hermes-agent.git
cd hermes-agent
pip install -e .
hermes setup   # 走初始化流程
```

### 第二步：恢复数据

```bash
# 停服
hermes stop  2>/dev/null

# 解压备份到 ~/.hermes（包含 L1+L3+配置+Skills）
tar xzf "hermes-backup-20260513_XXXXXX.tar.gz" -C ~/

# 恢复 L2 向量库（如果备份包里有 ~/.pg0/）
# 注意：L2 数据 153MB，解压后需要重启 hindsight 才能用
# 如果 hindsight 无法自动连接，手动：
cd ~/.hermes/hermes-agent && source .venv/bin/activate
nohup hindsight-api --port 9177 --idle-timeout 600 > /dev/null 2>&1 &

# 重启主程序
hermes start
```

### 第三步：重启后测试

```
（用微信或CLI跟Hermes说话）
→ 检测内部记忆是否还在（回答投资问题时带上持仓细节）
→ 检查Skills：hermes skill list → 应看到daily-briefing等
→ 检查Cron：hermes cron list → 应看到所有定时任务
→ 问一个历史问题："我们之前关于电池ETF讨论过什么？"
→ 如果上述都正常，迁移成功 ✅
```

## 5️⃣ 如果只丢失了特定层级

### 第1层（内部记忆）丢失
→ 恢复 `state.db`（内部记忆存在SQLite数据库里）
→ 如果state.db也没了：用 `external_memory/L1-内部记忆备份.md` 恢复。读那个文件，用 `memory(add)` 逐条写回。重建后需重启会话让L1生效。

### 第2层（Hindsight向量库）丢失
→ 恢复 `~/.pg0/` 整个目录（153MB），重启 hindsight 服务
→ 如果 `.pg0/` 也没了：从 L3 文件重建。读 `00-技能目录.md` → 读各skill → 用 `hindsight_retain` 逐条重建导航记忆（抖音监控/cookie提取/读知乎等）。语义搜索能力需从零积累，但结构化导航可恢复。详见下方「极限方案：只有L3，重建L1+L2」。

### 第3层（追踪文件）丢失
→ 恢复对应positions/*.md、events/*.md文件
→ 如果没备份：L3的05.md有持仓摘要，能重建大部分

---

→ 如果也没备份：第2层的05.md有摘要，能重建大部分

### Skills丢失
→ 恢复 `skills/` 文件夹
→ 如果没备份：需重新创建，但Skills代码通常也能从GitHub或其他方式找回

## 6️⃣ 反脆弱设计说明

本记忆系统在设计时已考虑了健壮性和冗余：

**1. 三层架构互为备份**
- 第1层丢失 → 第2层+第3层可重建
- 第2层丢失 → 第1层+历史会话可重建
- 第3层丢失 → 第2层有摘要

**2. 关键信息多副本**
- API密钥：config.yaml + 安全的地方
- 持仓规则：内部记忆 + 05.md + 第3层文件
- 监控条件：内部记忆 + daily-briefing skill + 05.md
- 定时任务：cron/jobs.json + 03.md（有手动重新创建说明）

**3. 原子化存储**
- 即使一个文件损坏，不影响其他文件
- 第3层文件独立存储，一个标的的文件坏了不影响其他标的

## 8️⃣ 极限方案：只有L3文件，重建L1和L2

> 适用场景：硬盘坏了、换电脑、重装系统，只有 `external_memory/` 和 `skills/` 保住了。

### 重建L1（内部记忆）

```bash
# 1. 新Hermes装好启动后
# 2. 读 external_memory/L1-内部记忆备份.md（包含全部18条规则）
# 3. 用 memory(add) 逐条写回，一条一条来
# 4. 写完后重启会话，L1自动注入每轮system prompt
```

```python
# 重建脚本思路（手动执行，非自动化）
# 读 L1-内部记忆备份.md 的代码块
# 对每条用 memory(action="add", content="...", target="memory") 写入
```

### 重建L2（Hindsight向量库导航）

```bash
# 1. 读 00-技能目录.md → 知道有哪些技能可用
# 2. 读 daily-briefing skill → 用 hindsight_retain 存"抖音监控"导航
# 3. 读 zhihu-reader skill → 存"知乎读帖"导航
# 4. 读 local-chrome-cdp-bridge skill → 存"cookie提取"导航
# 5. 读 topic-monitor skill → 存"话题监控编排"导航
# 6. 读 00-网站内容抓取总纲.md → 存"爬站三件套"导航
# 7. 读 00-宏观传导框架.md → 存"宏观框架"导航
```

重建后L2只能存结构化导航，语义检索能力需从零积累（约2-3天的正常使用量）。

### 重建后的能力对比

| 能力 | 恢复程度 | 说明 |
|:----|:--------|:-----|
| 知道你谁、持仓、工作流 | ✅ 完整 | L3保存了全部 |
| 定时任务 | ✅ 完整 | skills/ + external_memory/ 有记录 |
| 读知乎/抖音监控 | ✅ 完整 | skills/ + 脚本全在 |
| 出日报/周报 | ✅ 完整 | skills/ + 外部记忆全在 |
| L1自动导航（不绕路） | ✅ 完整 | L1-内部记忆备份.md 重建 |
| L2向量搜索（模糊联想） | ❌ 从零 | 语义索引需重新积累 |
| 历史会话（session_search） | ❌ 丢失 | 取决于state.db是否备份 |

**一句话总结：存好 external_memory/ + skills/ + L1-内部记忆备份.md = L1+L3全活，L2导航可重建，只是语义检索需重新积累。**
