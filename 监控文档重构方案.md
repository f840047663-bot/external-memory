# 监控文档重构方案

> **日期：** 2026-06-02
> **目标：** 解决"模型注意力漂移/像智障"问题
> **核心诊断：** 统筹框架太薄(只有路由表)，刻下来太厚(1015行全集)，子模块不完整(关键规则散落)→ AI要读10个文档才能跑完一步

---

## 一、根因诊断

### 1.1 信息散落在10+文件，执行一步需要跨5-7个文件

AI跑"阶段一-抖音-付鹏"这一个动作，实际需要读：
1. 统筹框架 → 知道调阶段一
2. 阶段一 → 知道三平台交替、失败重试逻辑
3. 抖音子模块 → 知道API格式
4. 决策树排错 → 知道失败怎么处理
5. 刻下来 → 发现决策树细节其实在这里更完整
6. 环境准备 → 发现Cookie判定逻辑
7. INDEX.md → 拿sec_uid

**结论：拆得太碎，每个子模块不自包含，统筹框架不内嵌核心规则。**

### 1.2 刻下来(1015行)是"全集"，子模块是"残集"

对比刻下来 vs 现有子模块，**刻下来独有的有效信息仅约100行**（~10%），其余900行是重复：

### 1.3 关键矛盾：断点更新时机说反了

| 来源 | 说法 | 行号 |
|:----|:----|:----|
| **刻下来** (2026-06-01用户纠正) | 下载+转录完成后**立刻更新** | L912-921 |

**刻下来标注了"2026-06-01用户纠正"，应以刻下来为准。** 但正因为矛盾存在于不同文件，AI读到哪个算哪个，必然出错。

### 1.4 统筹框架"大脑"只有路由表，没有决策能力

统筹框架121行，但核心内容只有：
- 路由表（7行）：告诉AI"调哪个子模块"
- 状态文件格式（8行）：JSON schema
- 执行规则（7行）：很笼统

**缺失的核心规则：**
- 决策树（失败怎么处理）→ 在决策树排错.md
- Cookie判定逻辑（什么时候刷新）→ 在环境准备.md
- 三平台交替详细规则 → 在阶段一.md
- 断点更新时机 → 在刻下来（而且说反了）
- 调试模式铁律 → 在阶段一.md

**结果：统筹框架名不副实，AI读了它还是不知道"遇到失败怎么办"。**

## 二、重构方案

### 2.1 设计原则

```
统筹框架 = 自包含的大脑（读它就能跑99%场景）
子模块   = 纯粹的手脚（只写平台特有细节）
刻下来   = 归档历史（不再作为执行依据）
INDEX    = 纯数据源（保持不动）
```

**关键转变：**
- 旧架构：统筹框架说"调子模块X" → 子模块说"失败看决策树" → 决策树说"Cookie问题看环境准备"
- 新架构：统筹框架内嵌决策树+失败处理+Cookie判定 → 子模块只说"这个平台的API格式和坑"

### 2.2 新文档架构

```
external_memory/
├── 监控统筹框架.md          ← 重写，~400行，自包含入口
├── 工作流模块/
│   ├── 抓取/
│   │   ├── 抖音.md          ← 精简，~50行，只留API+坑
│   │   ├── B站.md           ← 精简，~50行，只留API+坑
│   │   └── 知乎.md          ← 精简，~50行，只留API+坑
│   ├── 浏览器/
│   │   └── 兜底.md          ← 保留，~60行，删重复
│   └── 通用/
│       └── 环境准备.md      ← 保留，~80行，删重复
├── thinkers/
│   └── INDEX.md             ← 不动
├── archive/
│   └── 监控全流程-刻下来-历史版.md  ← 移入归档
├── 监控文档重构方案.md       ← 本文件
└── 监控处理断点.md           ← 不动（运行时数据）
```

**删除的子模块（内容合并入统筹框架）：**

**400行会不会太长？**
- 不会导致注意力漂移，因为AI是**按阶段顺序读**的：启动时读§0+§决策树(~120行进入上下文)，跑到阶段几就读§阶段几，之前的自然滑出窗口
- 远优于"读7行路由→跳5个文件→发现不够→再跳刻下来"的碎片化模式
- 用户担心的>500行漂移阈值，400行在安全范围内

### Step 4: 精简三个平台子模块
删除失败处理表、状态清单格式、Cookie逻辑。只留API格式+平台特有坑。

### Step 5: 精简浏览器兜底和环境准备
删除流程控制逻辑（已入统筹框架），保留操作命令。

### Step 6: 验证
用干净上下文测试：只给AI"监控统筹框架.md"，让它跑一轮，看是否能独立完成99%场景。

## 五、风险与缓解

| 风险 | 缓解 |
|:----|:----|
| 统筹框架400行是否仍会漂移 | 按§顺序读，每次只激活当前阶段内容；400行<500行安全阈值 |
| 用户习惯刻下来的完整叙述 | 归档文件保留，可随时查阅；统筹框架更精简高效 |
| 子模块太薄导致平台细节丢失 | 子模块保留所有"平台独有"内容（API格式、踩坑记录、特殊参数） |
| 未来新增平台 | 新增一个子模块文件即可，统筹框架不需要改 |

## 六、总结

**一句话：把"大脑"真正做大，把"手脚"真正做薄，把"刻下来"真正归档。**

- **统筹框架**从121行路由表 → 400行自包含执行手册（内嵌决策树、失败处理、Cookie逻辑、各阶段核心规则）
- **子模块**从70-100行（含大量重复） → 50行纯平台细节
- **刻下来**从执行依据 → 历史归档，不再参与执行
- **消除**所有信息重复和矛盾（特别是断点更新时机）
- **AI执行路径**从"读7个文件~1500行" → "读1个文件~200行+必要时跳1个子模块~50行"
