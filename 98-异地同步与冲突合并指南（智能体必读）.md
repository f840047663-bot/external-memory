# 异地同步与冲突合并指南（智能体必读）

> 本文档供任何新智能体（克隆盘B或恢复盘）阅读，重建与外部记忆仓库的完整同步工作流。

---

## 一、架构总览

```
┌────────────────────┐        ┌────────────────────┐
│  主盘 A (在线)      │        │  克隆盘 B (备用)    │
│  Hermes Agent       │        │  Hermes Agent      │
│  ~/.hermes/         │        │  ~/.hermes/         │
│  external_memory/   │        │  external_memory/   │
│         │           │        │         │           │
│   自动commit         │        │   自动commit         │
│   + push ↓          │        │   + push ↓          │
└────────┼────────────┘        └────────┼────────────┘
         │                               │
         ▼                               ▼
┌───────────────────────────────────────────────┐
│           GitHub 私有仓库（中央枢纽）            │
│   github.com/f840047663-bot/external-memory   │
│                                               │
│   A推上去的最新版 → B拉下来                      │
│   B改完推上去 → A再拉回来                        │
│   两台永远不会同时在线，全靠GitHub中转            │
└───────────────────────────────────────────────┘
```

### 核心设计原则

1. **GitHub是唯一可靠的中央枢纽** — A和B不需要同时在线，各自推送到GitHub，各自从GitHub拉取最新版
2. **本地仓库只做版本记录，不删** — 即使GitHub挂了，每台机器上的本地git历史仍然完整
3. **自动commit每30分钟** — 减少手动操作，确保记忆不丢失
4. **文件按日期/资产拆分天然抗冲突** — 大部分情况下不会产生冲突

## 二、首次搭建（新智能体第一次运行）

如果你是克隆盘B上首次运行的智能体，执行以下步骤：

### Step 1: 克隆仓库

```bash
# 如果 ~/.hermes/external_memory/ 还不存在
cd ~/.hermes/
git clone https://github.com/f840047663-bot/external-memory.git
```

> ⚠️ 如果克隆提示需要认证，用已保存的token：
> ```bash
> # token 存储在 ~/.hermes/config/github_token.txt
> TOKEN=$(cat ~/.hermes/config/github_token.txt)
> git clone https://f840047663-bot:$TOKEN@github.com/f840047663-bot/external-memory.git
> ```

### Step 2: 配置自动同步

```bash
# 设置git凭据（免交互push/pull）
echo "https://f840047663-bot:$(cat ~/.hermes/config/github_token.txt)@github.com" > ~/.git-credentials
chmod 600 ~/.git-credentials
git config --global credential.helper store

# 添加cron自动任务
(crontab -l 2>/dev/null; echo "*/30 * * * * cd ~/.hermes/external_memory && git pull --rebase --autostash -q origin master 2>/dev/null && echo 'pulled ok' || echo 'pull failed'") | crontab -
```

### Step 3: 检查可用token

如果token不存在或已过期，通过浏览器创建一个新的：
- 导航到 `https://github.com/settings/tokens/new`
- Note: `hermes-external-memory-sync`
- Scopes: 勾选 `repo`（Full control of private repositories）
- 点击 Generate token
- 将显示的token保存到 `~/.hermes/config/github_token.txt`

## 三、日常同步工作流

### 本地修改后推送到GitHub

```bash
cd ~/.hermes/external_memory
git add -A
git diff --cached --quiet || git commit -m "auto: $(date '+%Y-%m-%d:%H:%M')"
git push -q origin master
```

> 自动cron每30分钟执行上述操作，一般不需要手动执行。

### 获取远程最新修改

```bash
cd ~/.hermes/external_memory
git pull --rebase --autostash origin master
```

> `--rebase` 确保你的本地commit放在远程commit之后，保持线性历史。
> `--autostash` 自动暂存未commit的本地修改，拉取完再恢复。

### 手动强制覆盖（如果冲突太多）

```bash
# 放弃所有本地修改，完全同步远程版本
cd ~/.hermes/external_memory
git fetch origin
git reset --hard origin/master
```

> ⚠️ 慎用：会丢失所有本地未push的修改。

## 四、冲突处理（关键）

### 什么时候会产生冲突？

两个智能体同时修改了**同一个文件的同一段内容**并且都commit了。在 `git pull` 时会产生合并冲突。

典型冲突场景：
- `positions/*.md` — 两边的持仓分析同时改了同一持仓
- `INDEX.md` — 两边都修改了目录结构

### 冲突文件长什么样？

当发生冲突时，有冲突的 `git pull` 会报错：

```
Auto-merging positions/黄金.md
CONFLICT (content): Merge conflict in positions/黄金.md
Automatic merge failed; fix conflicts and then commit the result.
```

冲突文件内部会显示：

```
<<<<<<< HEAD
（这边的修改）
=======
（远程那边的修改）
>>>>>>> master
```

### 智能体处理冲突流程

```bash
# Step 1: 发现冲突后，先看看哪些文件有冲突
git status
# 会显示 "both modified: positions/黄金.md"

# Step 2: 检查冲突内容
# 用 read_file 工具读取冲突文件
# 看到 <<<<<<< HEAD 标记的段落

# Step 3: 决策逻辑
```

### 冲突决策逻辑（智能体版）

当遇到冲突时，按以下优先级判断保留哪个版本：

| 优先级 | 判断依据 | 示例 |
|:------|:---------|:-----|
| 1️⃣ **时间最新** | 保留日期更新的内容 | 都是价格数据 → 取日期最新的 |
| 2️⃣ **数据优先** | 保留数据更完整的内容 | A写了3个持仓理由，B写了5个 → 保留B的 |
| 3️⃣ **逻辑优先** | 保留分析更深入的内容 | A说"涨了"，B说"涨了因为美元弱" → 保留B的 |
| 4️⃣ **合并两者** | 把两边的合理内容拼起来 | A的前半段 + B的后半段 |
| 5️⃣ **相同内容去重** | 两边写了同样的观点 → 只保留一份 | 删除重复部分 |

### 冲突修复命令

```bash
# 方式一：手动编辑后标记已解决
# 读取文件 → 编辑 → 
git add <文件名>
git commit -m "merge: resolved conflict in <文件名>"

# 方式二：全盘接受远程版本（放弃本地修改）
git checkout --theirs <文件名>
git add <文件名>
git commit -m "merge: accepted remote version for <文件名>"

# 方式三：全盘接受本地版本（放弃远程修改）
git checkout --ours <文件名>
git add <文件名>
git commit -m "merge: accepted local version for <文件名>"
```

### 降低冲突风险的建议

1. **各写各的文件** — 如果你改了A文件，B智能体也改A文件，冲突概率高。尽量分工明确
2. **文件拆分足够细** — 每个资产单独一个文件，而不是所有分析写在一起
3. **push前先pull** — `git pull --rebase` 确保你是在最新版本上修改
4. **cron的auto-commit只含确认无误的内容** — 避免把半成品推上去

## 五、合并出错的回滚方案

合并错了完全不用担心，Git保留全部历史。以下三种方法按推荐程度排列：

### 方法1：还没push，直接中止合并（最安全）

```bash
git merge --abort
```
回到合并前的状态，跟没发生过一样。**只要还没 `git push`，随时可以这么干。**

### 方法2：已经push了，用revert撤销（推荐）

```bash
# 查看合并历史，找到那次合并commit
git log --oneline --graph -10

# 撤销合并（-m 1 表示保留第一父分支的内容）
git revert -m 1 HEAD

# 推送到远程
git push
```
这会生成一个"撤销合并"的新commit，**不删历史，不丢数据**，在远程也是干净的可追溯记录。多人协作场景最安全。

### 方法3：reset硬退回（暴力但是干净）

```bash
# 找到合并前的commit id
git log --oneline -10

# 硬退回到那个commit
git reset --hard <commit_id>

# 强制推送（--force-with-lease 比 --force 更安全）
git push --force-with-lease
```
用 `--force-with-lease` 而不是 `--force`，它会检查远程有没有别人推的新内容，有就拒绝，避免覆盖别人的工作。

### 方法4：只恢复某个文件到之前版本

```bash
# 恢复到上一个commit的版本
git checkout HEAD~1 -- positions/黄金.md

# 或者恢复到指定commit的版本
git checkout <commit_id> -- positions/黄金.md
```

## 六、克隆盘B的完整搭建流程（反脆弱设计）

### 核心原则

> **克隆盘B不能依赖A盘活着才能搭建。** 以下路径二选一：
> - **U盘离线模式**（反脆弱）：把所需文件从A拷到U盘 → U盘插B → B直接离线操作
> - **GitHub在线模式**（便利）：A活着的时候推送完，B直接clone

### 你需要物理拷贝到B盘的"三件套"

```
┌───────────────────────────────────────────────┐
│  U盘或新硬盘上的 "hermes-setup" 文件夹          │
│                                                 │
│  ①  github_token.txt  ← GitHub访问密钥          │
│  ②  本指南的PDF/打印版                           │
│  ③  （可选）完整 external_memory/ 离线备份        │
└───────────────────────────────────────────────┘
```

### 路径一：离线U盘安装（推荐，最反脆弱）

A盘上操作：

```bash
# Step 1: 在A盘把setup包准备好
mkdir -p /media/U盘/hermes-setup/

# Step 2: 拷贝token（没有这个，私有仓库clone不了）
cp ~/.hermes/config/github_token.txt /media/U盘/hermes-setup/

# Step 3: 可选——把整个仓库的最新版也拷过去（防止GitHub挂了）
cp -r ~/.hermes/external_memory /media/U盘/hermes-setup/
```

然后把U盘插到B盘，在B盘上：

```bash
# Step 1: 把token拷到B盘
mkdir -p ~/.hermes/config/
cp /media/U盘/hermes-setup/github_token.txt ~/.hermes/config/
chmod 600 ~/.hermes/config/github_token.txt

# Step 2: 克隆仓库（需要token）
cd ~/.hermes/
TOKEN=$(cat ~/.hermes/config/github_token.txt)
git clone https://f840047663-bot:$TOKEN@github.com/f840047663-bot/external-memory.git

# Step 3: 配置git credential（以后免密码）
echo "https://f840047663-bot:$TOKEN@github.com" > ~/.git-credentials
chmod 600 ~/.git-credentials
git config --global credential.helper store

# Step 4: 设置cron自动拉取（每30分钟检查更新）
(crontab -l 2>/dev/null | grep -v external_memory; \
 echo "# External memory: auto-pull every 30 minutes"; \
 echo "*/30 * * * * cd ~/.hermes/external_memory && git pull --rebase --autostash -q origin master 2>&1 | logger -t external-memory") \
 | crontab -

# Step 5: 验证
cd ~/.hermes/external_memory && git log --oneline -3
```

### 路径二：在线直接clone（需要A盘先推了最新版）

```bash
# Step 0: 需要先拿到token文件（拷U盘或scp/微信传文件）
# Step 1: clone
TOKEN=$(cat ~/.hermes/config/github_token.txt)
cd ~/.hermes/
git clone https://f840047663-bot:$TOKEN@github.com/f840047663-bot/external-memory.git

# Step 2: 配置cron（同上路径一Step 3-4）
```

### 第一次拉取之后

克隆完成后，B盘与普通仓库无异。B盘的cron设置的是 `git pull`（拉取），和A盘的 `git add → commit → push` 形成闭环：

```
A: 每30分钟 → commit → push ↗️
                              GitHub
B: 每30分钟 → pull          ↙️
```

> ⚠️ 重要：A和B的cron都是每30分钟执行一次。A改了内容，最多等30分钟就推上去了；B最多等30分钟就拉下来了。A和B**不需要同时在线**。

## 八、重要提示

1. **token安全** — `~/.git-credentials` 和 `~/.hermes/config/github_token.txt` 包含GitHub token，不要提交到仓库或分享
2. **token过期** — GitHub classic PAT到期后 `git push` 会报401错误

   **手动创建永久token（30秒，需要你在Chrome操作）：**
   1. 打开浏览器 → 访问 `https://github.com/settings/tokens/new`
   2. Note填：`hermes-external-memory-push`
   3. 过期时间选：**No expiration**（不选的话默认30天）
   4. Scopes勾选：**repo**（Full control of private repositories）
   5. 点击 Generate token → 复制显示的token
   6. 把token内容发给我，或者存到 `~/.hermes/config/github_token.txt`
   7. 我会自动更新配置

   **A盘已设每周一9点自动检查token是否失效，如果失效会在系统日志留下提示。**
   **⚠️ token换了记得告诉我，我要重新发邮件和更新本地配置。**

3. **不要同时push** — 两台机器同时push可能导致冲突。但由于A和B不会同时在线，这几乎不会发生
4. **cron失败会静默** — `*/30` cron每半小时执行一次，如果网络不通，下次还会再试。网络恢复后自动续上
5. **磁盘占用** — git仓库带完整历史，约占用50-100MB磁盘空间，在320GB硬盘上完全可忽略
