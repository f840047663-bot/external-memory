# 贝叶斯链条+日报/周报/月报系统 — 综合优化设计方案

> 生成时间：2026-05-18
> 适用背景：¥15万总资金，22项持仓，14个抖音博主监控，Hermes Agent运行

---

## 〇、核心设计原则（重新梳理）

用户核心公式：
```
P(后验) = P(先验) × P(新证据|先验) / P(新证据)
仓位分配 = f(期望收益) = 上行空间 × P(上行|全部证据) - 下行空间 × P(下行|全部证据)
```

**三层约束（优先级从高到低）：**
1. **事实理解** — 系统帮用户理解「发生了什么」「逻辑链怎么走的」，不是替用户做决策
2. **完整追溯** — 每条更新可查：事件→推理→概率变化→验证标记
3. **可持续** — token成本可控，22项链+3份报告不炸预算

## 一、贝叶斯链的资产覆盖策略

### Q1: 22项持仓全部维护独立贝叶斯链？还是只维护核心资产的？

**建议：三级覆盖，不是一刀切**

| 级别 | 条件 | 数量预估 | 更新方式 | Token成本 |
|:----|:-----|:--------|:--------|:---------|
| 🔴 **完整贝叶斯链** | 仓位>5% 或 核心逻辑活跃（芯片/恒生科技/养殖/黄金/电池） | 6-8项 | 每次有新信息就更新 | ~200-400 token/次 |
| 🟡 **简化链**（仅概率标签+仓位逻辑） | 仓位1-5%或被动持有（有色金属/纳指/债券/绿色电力等） | 8-10项 | 每周一次批量更新 | ~50-100 token/次 |
| ⚪ **无链**（仅净值监控） | 蚊子仓<1%或现金等价物 | 4-6项 | 只在周报中统计分布 | 0 token |

**WHY：**
- 完整链每条更新约200-400 token（含追溯字段），22项每天都更新 = 4,400-8,800 token/天≈¥0.01-0.02 → 看起来少，但加上日报/周报/月报的聚合推理成本，累积可观
- 多数持仓（纳指/债券/绿色电力）每周甚至每月才有新信息，维护每日链是浪费
- 简化链只记录「概率标签+关键事件」，不写完整推理过程，大幅降低token

**HOW：**
```python
BAYESIAN_CHAIN_CONFIG = {
    "full": ["芯片ETF_020628", "恒生科技_012349", "养殖ETF_014415", 
             "黄金_实物", "金银珠宝_002207", "电池ETF_018926", 
             "海外科技_501312"],  # 7项
    "lite": ["稀有金属_014110", "有色金属_004433", "纳指_017436",
             "绿色电力_017175", "永赢制造_018124", "自由现金流_024575",
             "鑫享添利_012951", "景颐债券_010011", "医药_015915"],  # 9项
    "none": ["碳中和_018419", "有色金属小_017141", "AI联接蚊仓", 
             "其他6个迷你仓"]  # 蚊子仓不计
}
```

### Q2: 概率更新用定性还是定量？

**建议：双层并行 — 定性做日常决策，定量做月底校准**

| 维度 | 定性（低/中/高） | 定量（百分比） |
|:----|:----------------|:--------------|
| **日常更新** | ✅ **主力** | ❌ 不用 |
| **月底校准** | ❌ 不够精细 | ✅ **主力** |
| **Token成本** | ~50 token/次 | ~80 token/次（含推理） |
| **认知负担** | 低，一眼看懂 | 中，需要锚定基准 |
| **可回测性** | 差，不能算准确率 | 好，可以算Brier分数 |

**WHY：**
- 用户原话强调「理解事实」而不是精确量化，日常使用定性足够
- 但回测准确率时需要定量——否则无法区分"60%对"和"90%对"
- 定性+定量的映射关系需要固定，否则每次判断标准不一致

**HOW：**
```
日常更新用定性：
  P(上行) = {极低 | 低 | 中低 | 中 | 中高 | 高 | 极高}
  对应仓位动作 = {清仓 | 减仓 | 不动 | 轻仓 | 持有 | 可加仓 | 加仓到上限}
  证据强度 = {弱(1x) | 中(2x) | 强(4x)}

月底校准转定量：
  定性→定量映射固定表：
    极高 = 85-95%
    高 = 70-85%
    中高 = 55-70%
    中 = 45-55%
    中低 = 30-45%
    低 = 15-30%
    极低 = 5-15%
  
  校准公式：Brier Score = 1/N ∑(P_pred - actual)²
  好的校准：Brier < 0.15
  需调整：Brier > 0.25
```

### Q3: 一个信息源影响多个资产时，怎么处理关联更新？

**建议：采用「信息源→资产映射表」一次性批量更新，共享推理成本**

**WHY：**
- 当前做法是逐个资产手动更新，每个都要重新读一遍信息源全文，重复消耗token
- 大牛洲的一个视频可能同时影响芯片/恒生科技/整体市场，一次推理可以产出所有受影响资产的更新

**HOW：**
```
信息源处理流程：
1. 转录/摘要 → 提取核心论点（一次推理，~300-500 token）
2. 分类器判断影响哪些资产（一次推理，~100 token）
3. 批量更新：对每个受影响资产做贝叶斯更新（共享步骤1的结果）
   - 步骤1的结果作为shared_context传入，不需要重新读原文
   - 每个资产更新只需要增量推理：~100-150 token/个

示例（大牛洲《AI三层蛋糕》视频）：
  共享推理（300 token）：
    提取论点：①科创50 PE 160倍 ②腾讯PE<16倍 ③黄仁勋应用层泡沫警告
    分类结果：影响芯片(+1证据)、恒生科技(+1证据)、整体市场(+1风险提示)
  
  资产更新：
    芯片（150 token）：P(下行)中→中高，逻辑：泡沫风险上升
    恒生科技（100 token）：P(上行)中→中高，逻辑：腾讯估值安全+AI叙事
    整体市场（50 token）：风险标签更新

  总成本：300+150+100+50 = 600 token
  对比独立处理：4×400 = 1600 token（节省62%）
```

### Q4: 初始先验怎么定？

**建议：建仓时设定一次，重大基本面变化时重设**

**初始先验设定方法：**
```
建仓时先验 = f(资产类型, 估值位置, 持仓成本)
  
  资产类型系数：
    战略配置（纳指/黄金）→ P(上行)=75%（默认偏高，因为长期持有信仰）
    战术逻辑（电池/芯片）→ P(上行)=60%（逻辑已验证但波动大）
    周期反转（养殖/银行）→ P(上行)=50%（纯等信号，方向未定）
    趋势投机（已清仓）→ 不设链
    对冲（金银珠宝C）→ P(上行)=50%（跟随金价，不独立判断）

  估值修正：
    低估值（如腾讯PE<16）→ +10%
    高估值（如科创50 PE 160）→ -10%
    合理估值 → 不修正

  持仓成本修正：
    成本在近一年低位 → +5%（安全垫厚）
    成本在近一年高位 → -5%（追高）

  重设条件（任一触发）：
    - 该资产的基本面驱动因素发生根本变化（如行业政策180度转弯）
    - 用户的投资框架做了重大调整
    - 该资产的持有逻辑完全改变（如从趋势投机重新分类为战术逻辑）
```

**COST：**
- 初始设定：每项一次，~150 token/项
- 22项一次性设定 ≈ 3,300 token（一次性）
- 重设：每年预估2-3次，成本忽略不计

## 二、更新频率与优先级

### Q5: 怎么判断哪些资产今天需要更新？

**建议：设「更新触发器」规则，不轮询不主动扫描**

```
更新判断流程（每日13:00博主动态处理后执行）:

1. 扫描今日所有信息源产出
   - 14个抖音博主新视频 → 提取各自核心论点
   - 财联社电报 → 提取重大事件（仅限直接影响持仓的）
   
2. 对每个信息源做「资产影响分类」（共享推理，见Q3）
   - 输出：{信息源ID, 受影响资产列表, 证据强度}
   
3. 有受影响资产的 → 今日需要更新该资产贝叶斯链
   无受影响资产的 → 跳过，先验不变

4. 额外触发器（无论是否有新信息）：
   - 单个资产单日净值波动 > ±4% → 强制归因，即使博主没提及
   - 重大宏观事件（美联储决议/中美峰会等）→ 批量扫描所有受影响资产
```

**WHY：**
- 不是每天22项都有新信息，强行「假装更新」是伪工作，消耗token还误导用户
- 14个博主每天发视频，平均每个视频影响1-3项资产 → 每日实际更新数 ≈ 5-10项（含重复）
- 被动等待信息+判断是否相关，比主动扫描22项问「你有新信息吗」高效得多

**COST：**
- 每日信息影响分类：~300-500 token（共享一次，产出所有受影响资产列表）
- 平均每日5-10个资产更新：~500-1000 token
- 日总成本（不含日报）：~800-1500 token ≈ ¥0.001-0.003（DeepSeek V4 Flash）

### Q6: 日/周/月不同时间粒度各更新什么？

**建议：三层时间粒度，每层有不同的视角和Token预算**

```
┌─────────────────────────────────────────────┐
│              月报                            │
│  置信度校准 + 基本面重评估 + 框架验证        │
│  预算：~3000-5000 token                     │
│  时间：每月最后一周日                          │
├─────────────────────────────────────────────┤
│              周报                            │
│  趋势判断 + 多源共识 + 机会成本排名更新      │
│  预算：~2000-3000 token                     │
│  时间：每周日 20:00                          │
├─────────────────────────────────────────────┤
│              日报                            │
│  净值变动 + 当天博主观点 + 今日贝叶斯更新    │
│  预算：~800-1500 token + 链更新成本          │
│  时间：每日 15:30                            │
└─────────────────────────────────────────────┘

【日报】每日15:30 — 预算~800-1500 token
  🅰 净值速览（API直拉，0 token）
     - 总净值变动 + 涨跌幅前3/后3
     - T0核心资产当日P&L
  
  🅱 今日信源陈列（从13:00聚合结果提取，0额外token）
     - 博主新观点摘要（每条1行）
     - 财联社关键事件（仅影响持仓的）
  
  🅲 今日贝叶斯更新区块「新增结构」— ~300-500 token
     - 今日有更新的资产列表（无更新则跳过）
     - 每条格式：
       【资产名】P(上行): 中→中高 | 证据: 大牛洲说腾讯PE<16 | 仓位影响: 等右侧策略不变 | 待验证: 下个博主是否共识

【周报】每周日20:00 — 预算~2000-3000 token
  🅰 本周P&L（API直拉，0 token）
  🅱 叙事串联（宏观→中观→微观三层框架）
     - 宏观框架更新（中美/美联储/地缘）
     - 板块叙事变化（延续/调整/转向）
     - 每项关键资产新闻链（日期|净值|事件）
  
  🅲 贝叶斯链周汇总「新增结构」— ~500-800 token
     - 每项活跃资产的P(上行)周变动趋势
     - 证据强度累计统计（本周新增证据条数/来源分布）
     - 置信度变化汇总（↑/→/↓ 各几项）
  
  🅳 机会成本排名更新
     - 跨资产比较排序
     - 仓位倾斜建议

【月报】每月最后周日 — 预算~3000-5000 token
  🅰 月度P&L汇总
  🅱 贝叶斯置信度校准「核心新增」
     - 对比月初预测和月末实际走势
     - Brier Score计算（定量校准）
     - 各资产「先验→后验→实际」三元对照表
     - 哪些判断准确？哪些偏差大？
  
  🅲 信源质量评估
     - 各博主的「历史准确率」统计
     - 哪些博主该降权/升权？
  
  🅳 框架重校准
     - 宏观框架是否需要调整？
     - 持仓分类是否需要重新划分？
```

## 三、报告结构优化

### Q7: 日报格式怎么跟贝叶斯链挂钩？

**建议：日报末尾加「今日贝叶斯更新」区块，无更新就跳过**

```
━━ 今日贝叶斯更新 ━━

🔄 有更新（3项）：
  【芯片 020628】P(上行): 中高→中 | 证据: 大牛洲黄仁勋泡沫警告(-1级) | 仓位: 等回调策略不变 | 验证标记: ⏳待下一博主验证
  【恒生科技 012349】P(上行): 中→中高 | 证据: 大牛洲腾讯PE<16安全边际(+1级) | 仓位: 等右侧不变 | 验证标记: ✅已交叉验证(腾讯基本面数据吻合)
  【整体市场】风险标签: 正常→偏高 | 证据: 估值泡沫信号增加 | 验证标记: 📌月度校准

➡️ 无更新（4项）: 养殖/电池/金银珠宝/海外科技 — 今日无相关信息

━━━━━━━━━━━━━━━━
```

**WHY：**
- 日报的核心功能是「帮用户节省时间」，贝叶斯链区块是让用户一眼看到「今天什么变了」
- 无更新就跳过，避免空转token
- 每条更新包含验证标记，方便用户评估更新质量
- 紧凑格式控制在200-400 token以内

### Q8: 周报中如何汇总一周贝叶斯链变化？

**建议：每个活跃资产一张「周趋势表」，再加全局汇总**

```
━━━ 贝叶斯链周汇总 ━━━

【P(上行)周变动】

芯片 020628    中→中高→中→中          ↕↕↕  平均: 中 （波动频繁，受泡沫/国产替代双线拉扯）
恒生科技      中→中→中高→中高        ↗↗↗  平均: 中高（腾讯估值获得共识支撑）
养殖 014415   中→中→中→中            →→→  平均: 中 （周期底部未变）
电池 018926   中高→中高→中高→中高    →→→  平均: 中高（逻辑稳定）
金银珠宝      中→中→中→中            →→→  平均: 中 （跟随金价，本周无新信息）

【证据统计】本周新增有效证据：
- 来源分布：大牛洲2条 | 付鹏1条 | 伊娃1条 | 财联社2条
- 证据强度：强3条 | 中2条 | 弱1条

【置信度变化汇总】
↑ 上调（逻辑增强）：恒生科技（腾讯估值+AI叙事）
→ 不变：养殖/电池/金银珠宝/海外科技
↓ 下调（风险上升）：芯片（泡沫风险警告）
```

**WHY：**
- 用户需要「一眼看明白一周的趋势」，不是逐条读更新日志
- 趋势表比数字列表更直观（↗↘→符号成本几乎为零，人类视觉识别快10倍）
- 证据统计让用户知道「这周的信息质量如何」
- 所有内容来自日报中积累的贝叶斯更新，不需要额外推理

**COST：**
- 周汇总推理成本：~300-400 token（将一周的日更新聚合成趋势表）
- 因为每日更新已经存储了结构化数据，聚合只是格式化

### Q9: 月报做置信度校准

**建议：月度校准是贝叶斯闭环的关键 —— 不校准=盲人摸象**

```
━━━ 月度置信度校准 ━━━

【关于校准的说明】
月校准不是「预测未来比赛」，而是帮用户找出认知偏差：
- 如果总是过度自信（P=80%实际只对了60%）→ 需要降权所有判断
- 如果总是过于保守（P=55%实际对了80%）→ 可以更信任自己的判断
- 校准的目的是「让你的概率估计更贴近真实世界」，不是追求准确率

【校准方法】
使用简化的定性校准（避免复杂数学）：
1. 对每项资产，取出月初的 P(上行) 定性值
2. 对比月末实际走势：上涨/下跌/横盘
3. 评估判断方向是否正确（方向准确率）
4. 评估置信度是否合理（过度自信/保守倾向）

【校准表格式】

| 资产 | 月初P(上行) | 月末走势 | 方向判断 | 置信度评估 |
|:----|:-----------|:--------|:--------|:----------|
| 芯片 | 中高(70%) | ↓-4.3% | ❌误判 | ⚠️过度自信 |
| 恒生科技 | 中(50%) | →+0.8% | ✅中性正确 | ✅合理 |
| 养殖 | 中(50%) | →-0.5% | ✅中性正确 | ✅合理 |
| 电池 | 中高(70%) | ↑+3.1% | ✅方向正确 | ✅合理 |
| 金银珠宝 | 中(50%) | →+1.2% | ✅中性正确 | ✅合理 |

【方向准确率】4/5 = 80%（芯片误判降低总体）
【置信度偏差】1项过度自信（芯片），0项过于保守
【校准建议】芯片的判断需要更谨慎，泡沫信号出现时应更快速降权

【信源质量评估】

| 信源 | 本月贡献证据数 | 验证正确 | 验证错误 | 待验证 | 建议 |
|:----|:-------------|:--------|:--------|:------|:----|
| 大牛洲 | 4条 | 3 | 0 | 1 | ✅可信度高，权重维持 |
| 付鹏 | 2条 | 2 | 0 | 0 | ✅可信 |
| 伊娃 | 3条 | 2 | 1 | 0 | ⚠️有一条偏差（养殖反转时间） |
| 财联社 | 6条事件 | — | — | — | 事实性信源，不评准确率 |
```

**WHY：**
- 没有校准的贝叶斯系统=没有反馈环，会越跑越偏
- 月度校准一次的成本约1500-2000 token，相比每日校准省10倍
- 重点是「校准趋势」而不是「追求单次准确」——偏差的方向比偏差的大小更重要

## 四、可持续性与成本控制

### Q10: 22项都维护链，token成本会增加多少？

**建议：按三级覆盖策略执行，年度总成本约¥30-50**

```
成本预估（基于DeepSeek V4 Flash计价：入¥0.5/百万token，出¥2/百万token）：

【每日】8项完整链 + 10项简化链
  信息影响分类：~400 token（共享推理）
  完整链更新(平均6项/天)：6 × 150 token = 900 token
  简化链更新(周一度批量)：0 token（非当天）
  日报贝叶斯区块：~200 token
  日小计：~1,500 token → ¥0.0015-0.003/天（输出占30%计¥0.0009-0.0018）

【每周】
  一周累积的日更新：5天 × 1,500 = 7,500 token
  周报贝叶斯汇总：~400 token
  周小计：~400 token → ¥0.0004-0.0008

【每月】
  月校准：~1,500 token → ¥0.0015-0.003
  月小计：~1,500 token

【年度总预估】
  基础日更新：1,500 × 250交易日 = 375,000 token
  周汇总：400 × 52周 = 20,800 token
  月校准：1,500 × 12月 = 18,000 token
  链历史文件读写（约）：10,000 token
  年度总token：~423,800 token
  年度总成本：~¥0.85-2.12（按输出占比30%估算）
  
  加上日报本身的成本（不包含在链成本中）：
  日报：~800-1,500 token/天 × 250天 = 200,000-375,000 token
  周报：~2,500 token/次 × 52周 = 130,000 token
  月报：~4,000 token/次 × 12月 = 48,000 token
  报告总token：~378,000-553,000 token → ¥0.76-2.77

【合计】贝叶斯链(~¥1-2) + 报告体系(~¥1-3) ≈ 年度¥2-5
```

**优化手段：**
1. **三级覆盖（最重要）** — 完整链6-8项，简化链8-10项，无链4-6项，减少50%+更新
2. **共享推理** — 一个信息源影响多个资产时共用上下文，节省30-60%
3. **结构化存储** — 日更新不是NLP全文，而是结构化JSON格式，读写token更低
4. **内容寻址** — 同一个博主说同一观点的第2次往后不计入（证据饱和），节省token
5. **离线归档** — 180天前旧链移出活跃存储

### Q12: 系统如何自我验证？

**建议：三重验证机制，从宏观到微观**

```
【验证层级一：月度置信度校准】（详见Q9）
  频率：每月一次
  内容：对比预测vs实际，计算准确率和偏差方向
  输出：校准报告+调整建议

【验证层级二：信源质量评分】（月度更新）
  频率：每月一次，在月校准中完成
  评分维度：
    - 方向准确率：博主判断方向是否正确（涨/跌/横盘）
    - 时效性：观点是否在合理时间窗内兑现
    - 逻辑性：论据是否经得起数据验证
  
  评分规则（累积法，不是逐条打分）：
    - 每条证据验证后更新该博主的「准确计数」和「偏差计数」
    - 月度输出「方向准确率 = 正确/(正确+偏差)」(不含待验证的)
    - 连续3个月准确率<50% → 降权（该博主更新只标记不更新链）
    - 连续3个月准确率>80% → 升权（优先纳入链更新）

  淘汰机制（季度触发）：
    如果某博主连续3个月准确率<40%且总贡献证据<6条 → 标记为「低质信源」
    低质信源的更新不再触发贝叶斯链更新，仅做日志记录

【验证层级三：框架压力测试】（季度一次）
  内容：检查宏观框架是否需要更新
  触发条件（任一）：
    - 连续2个月校准准确率<60%
    - 出现框架无法解释的反常事件（累计3次）
    - 用户主动提出框架调整
  
  操作：
    1. 导出最近一季度的所有贝叶斯更新记录
    2. 检查哪些资产的判断偏差最大
    3. 分析偏差的根因：是信源问题？框架问题？还是黑天鹅？
    4. 输出框架调整建议
```

**COST：**
- 月度置信度校准：~1,500 token（已计入Q10月报成本）
- 信源质量评分：并入月校准，0额外token
- 季度压力测试：~2,000-3,000 token（季度一次）
- 总成本：~¥0.01-0.02/年，几乎可忽略

## 六、关键风险与应对

| 风险 | 概率 | 影响 | 应对 |
|:----|:----|:----|:----|
| 链更新变成机械记录（用户最怕的） | 高 | 高 | 每条更新必须有「推理」字段，禁止只写「P(上行)中→中」没原因 |
| 证据饱和规则失效（重复更新） | 中 | 中 | 硬编码检测：同一博主同一资产同一论点到第2次就跳过 |
| 月度校准变成数字游戏 | 中 | 低 | 保持每月校准报告的人可读性，不追求数学精确 |
| 链文件膨胀 | 低 | 低 | 180天归档机制有效防止 |
| 用户改变框架（如重新分类资产） | 低 | 中 | 简易迁移脚本：从旧类重设先验+重设时间轴 |
