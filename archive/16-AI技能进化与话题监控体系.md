## 当前技能架构（三层解耦）

```
通用积木层（可替换、可复用）
├── cookie-extractor  — 从Chrome提cookie，任何平台通用
├── waf-bypass        — 手机UA绕过WAF封IP，任何平台通用
│
平台专用层（每个平台一个，依赖通用层）
├── zhihu-reader      — 读知乎帖子/回答/搜索
│
编排层（组合积木形成工作流）
└── topic-monitor     — 给定话题→搜索→读帖→分析→长期跟踪
```

### 设计原则

1. **每层做好一件事** — cookie提取不管知乎是什么，读知乎不管cookie怎么来的
2. **替换不掉链子** — cookie提取方式换了，zhihu-reader不用改
3. **新增低成本** — 想监控B站？建个bili-reader，直接复用cookie-extractor和waf-bypass
4. **任意层可独立升级** — 今天发现更好的WAF绕过方法，只改waf-bypass一个文件

## 正确使用大模型的方法（大好人的方法论）

> 大好人 2026-05-18 总结。**用编程思维玩大模型，而不是把AI当神灯。**

### 核心理念

大模型不是万能的。它不知道你的需求、不知道你的环境、不知道你的前置条件。正确用法：

**① 告诉他功能怎么实现**
不是让AI自己猜，而是告诉它实现路径："先提取cookie→再绕WAF→再读知乎"

**② 告诉他有哪些库/技能可以用**
像编程时告诉程序员"有requests库、有pandas"一样，告诉AI"你有cookie-extractor、waf-bypass、zhihu-reader这些技能"

**③ 告诉他架构和步骤**
三层解耦、编排层组合积木——架构是你定好的，AI负责执行。

### 跟传统编程的类比

| 传统编程 | AI Agent玩法 |
|---------|-------------|
| 程序员写代码 | 用户告诉AI怎么做 |
| 调用API/库 | 调用已存好的技能 |
| 架构师画架构图 | 用户定分层和依赖关系 |
| 单元测试 | 用户反馈纠偏 |
| 部署上线 | 技能生效，下次直接用 |

### 为什么这样更反脆弱？

1. **AI本身不可靠** — 今天DeepSeek明天Claude，同一个模型不同版本也可能行为不同
2. **用户知道环境** — 你知道你Chrome什么版本、知乎什么状态、IP有没有被封
3. **步骤写死了就不依赖AI判断** — "先A再B再C"是确定性流程，AI执行即可
4. **任何AI都能接盘** — 纯文本技能备份在L3，不依赖特定框架

### 总结一句话

> **把大模型当执行力，不当创造力。架构你定、步骤你定、工具你定，AI只负责跑通。**
