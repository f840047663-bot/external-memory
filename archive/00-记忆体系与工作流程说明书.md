### 什么是互锁

每个文件不是孤立存在的。文件内部用 `🔗 互锁→` 标记指向相关文件。**读任何一个文件时，顺着箭头能找全上下游。**

### 三个目录双向互锁

```
events/（事件链）
  ← 事件影响多个持仓，一次列全
  → 持仓引用事件时只写一句话+链接
       ↕
positions/（持仓文件）
  ← 每个事件的影响写在自己的事件链里
  → 🔗 互锁→ events/YYYY-MM-DD.md(事件名+影响方向)
       ↕
thinkers/（思想家档案）
  ← 博主观点→具体事件戳
  → 🔗 互锁→ events/ & positions/
```

### 关键原则：读一个文件=读到全部关联信息

```
错误做法（旧方案）：
  读 positions/chip.md → 看到🔗互锁→ thinkers/ → 再打开 thinkers/
  → 要读两个文件才能拼出完整画面 ❌

正确做法（新方案）：
  读 events/2026-05-19.md → 看到：
    "韩国闪崩→芯片🟡(情绪传导) | 海外科技🟡(估值联动)"
  → 一个文件读完，所有受影响持仓和逻辑链一目了然 ✅
```


### P(上行)↔买入触发联动规则

**核心原则：** 买入触发价不是固定的，它随贝叶斯概率联动：

```
触发价 = 当前净值 × (1 - 安全垫)
安全垫 = 基准安全垫 × (1 - P(上行))

P(上行)=80% → 安全垫极小 → 触发价接近现价（敢买）
P(上行)=50% → 安全垫=基准 → 触发价较低（需打折）
P(上行)=30% → 安全垫放大 → 触发价更低（需大幅打折）
```

**为什么这样设计：**
- P(上行)越高，确定性越强，不需要那么多安全边际
- P(上行)越低，不确定性越大，需要更大折扣来对冲
- 避免"P越来越高但触发价越来越低，永远买不到"的悖论

**硬条件独立于概率：**
某些硬条件（如"连续3周不创新低+放量30%"）是独立门槛，不随P变化。P再高也得满足硬条件才动手。

### 概率标注规则

**所有概率必须显式写 P(上行) 或 P(下行)，不能只写数字。**

```
✅ P(上行) 65%→57%（-8pp，贷款负值）
✅ P(下行) 30%→35%（+5pp，宽松预期减弱）
❌ 65%→57%
❌ P 65%→57%
```

### 三层互锁分工

| 层 | 存什么 | 如何工作 |
|:--|:-------|:---------|
| **第三层** | `🔗 互锁→ [文件名]（看哪节）` 的文字链接 | 源——完整关系网络，永不丢失 |
| **第二层** | 关键互锁的语义记忆 | 对话中自动召回（如"银行跌了"→自动关联古都闲云的贷款负值观点） |
| **第一层** | 一条路牌 | 强制引导不忘记看互锁 |

## 四、日常日报工作流程

### 数据流

```
每天 13:00
抖音监控 → 接口直调查最新视频 → 标题筛选(持仓相关才下载)
  → 命令行下视频 → 容器转文字
  → 深度求索分析 → 提取核心观点
  → 观点写入：events/{日期}.md（新事件链表格，列影响持仓）
  → 同时更新对应 thinker 档案的「最新观点」
  → 关键互锁写入：第二层后见之明

每天 15:30
助手做盈亏计算 → 事件归因 → 贝叶斯更新
  → 先读 events/ 看今天有什么新事件
  → 再读 positions/*.md 获取当前状态
  → 输出日报（格式见下方）
  → 发邮件到 feng202210062126@qq.com
```

### 事件→持仓映射流程

```
新观点进来
  ↓
判断影响哪些持仓（可能多个）
  ↓
写入 events/{日期}.md（完整的持仓影响表）
  ↓
每个被影响的持仓文件头部补一句：
  🔗 事件互锁→ events/{日期}.md(事件名→影响方向)
  ↓
读持仓 = 知道所有影响它的事件
读事件 = 知道所有被影响的持仓
双向往来，不会漏
```

### 日报结构

```
━━━ 5月X日 ━━━ 今天世界在发生什么 ━━━

🌍 宏观传导（自上往下）
[宏观事件 -> events/YYYY-MM-DD.md]
  → 影响：[传导路径]
  → 影响持仓：[列表]

━━━ 仓位分析 ━━━
📦 [资产名]
│ 事件链：events/YYYY-MM-DD.md引用
│ 📖 思考：推理过程
│ ├─ 上行空间：X%
│ ├─ 下行空间：X%
│ └─ 贝叶斯：概率(上行) X%→Y% | 仓位建议

━━━ 无更新 M项 ━━━
━━━ 值得盯的 ━━━
```

### 什么观点值得入库

| 条件 | 说明 |
|:----|:-----|
| ✅ 带时间预判 | 说了大概什么时候会发生什么 |
| ✅ 带价格/数据目标 | 说了具体数字 |
| ✅ 带因果关系/逻辑链 | 解释了为什么这么判断 |
| ✅ 与持仓相关 | 直接或间接影响持仓 |
| ❌ 纯预测无逻辑 | 只说"会涨"不说为什么，跳过 |
| ❌ 纯情绪喊口号 | 跳过 |
| ❌ 重复已有观点 | 不重复记录，只更新验证状态 |

### 事件入库流程

```
抖音/知乎新内容
  ↓ AI分析提取核心观点
  ↓ 判断：
    ① 这是一个新事件吗？→ 写进 events/{日期}.md
    ② 影响哪几个持仓？→ 事件链表格列全
    ③ 这个博主有什么长期观点？→ 更新对应 thinker 档案
    ④ 关键关联关系→ 写一条第二层记忆
  ↓
最终效果：
  events/ = 当天所有事件的全景地图
  positions/ = 每个持仓受影响的所有事件历史
  thinkers/ = 博主长期立场和预判
```

## 六、系统稳定性

### 卡死修复

第一反应跑：
```bash
bash ~/桌面/修复卡死.sh
```
根因始终是后见之明进程叠罗汉 → 内存耗尽 → 交换分区倒腾 → 磁盘输入输出卡死。

### 强模型兜底

搞不定的事（网站登录凭据解密/坑调试/未知领域）不要硬磕。
3次工具调用没结果 → 委托任务派更强模型（克劳德/生成式预训练变换器等）。

### 改配置铁律

改模型/配置前：
1. 全备份配置文档
2. 通知小龙虾
3. 网络请求测试通才改
4. 改完10分钟内互相发确认，超时=出事按备份恢复
