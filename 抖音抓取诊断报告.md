## 3. 之前CDP方案为什么失败

现有两个旧脚本都依赖CDP浏览器方案：

### 3.1 `monitor_fetch.py`（CDP方案）
- **方法：** `cdp-browser-reader.py` 导航到抖音博主主页 → 从HTML里提取`__INITIAL_STATE__` JSON → 解析视频列表
- **失败原因：**
  1. **CDP依赖浏览器进程** — Chrome需要以debug mode运行，经常挂了/重启后WS URL变化
  2. **CDP浏览器ID变化** — Chrome重启后`webSocketDebuggerUrl`变化，`config.yaml`里的CDP URL过时
  3. **抖音页面结构复杂** — `__INITIAL_STATE__` JSON可能不在首屏（需要滚动），CDP等待15秒不够
  4. **HTML解析脆弱** — 多个正则模式兜底（方式1→方式2→方式3），任何一个匹配失败就返回空
  5. **慢** — 每个博主需打开新页面+等待15秒渲染+解析，9个博主需约3分钟
- **monitoring_checkpoint.json记录：** 「抖音CDP全部空(页面结构问题/限流，按要求跳过不重试)」

### 3.2 `scrape_all_platforms.py`（旧综合脚本）
- **方法：** 旧版用curl + cookie调抖音List API
- **失败原因：**
  1. **限流（-799）** — 抖音API对curl/scrapy限制严格，没有yt-dlp的浏览器cookie环境
  2. **B站WBI签名问题** — 长期-403失效
  3. **单体黑箱** — 540行单体，不符合用户要求的解耦模式
  4. **被文档明确禁止使用** — SKILL.md §⛔ 必须走解耦模板

### 3.3 `monitor_fetch.sh`（shell版本）
- **方法：** curl直接调API
- **失败原因：** 同样是-799限流，没有yt-dlp的浏览器指纹

### 3.4 对比总结

| 方案 | 方法 | 状态 | 根因 |
|:----|:----|:----:|:-----|
| `douyin_monitor.py` ✅ | yt-dlp提取cookie → 调内部API | **工作正常** | yt-dlp自带浏览器指纹，不被限流 |
| `monitor_fetch.py` ❌ | CDP浏览器 → HTML解析 | **全部空** | CDP依赖浏览器进程，解析脆弱 |
| `scrape_all_platforms.py` ❌ | curl + cookie → 调API | **-799限流** | 无浏览器指纹，被抖音拦截 |
| `monitor_fetch.sh` ❌ | curl直调 | **-799限流** | 同上 |

**核心结论：** yt-dlp的`--cookies-from-browser chrome`是关键——它从Chrome的加密cookie存储中提取session，带完整的浏览器指纹。curl/cdp方案都没有这个指纹，所以被限流。
